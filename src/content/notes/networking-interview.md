# 计算机网络面试题集锦

## 1、什么是HTTP？HTTP和HTTPS的区别和联系是什么？

HTTP全称是：**Hyper Text Transfer Protocol**，即超文本传输协议，是Web端最基础的通信协议，它规定了**客户端和服务端之间如何请求和响应**。但是HTTP有一个致命的缺点：**它是明文传输的，极不安全**。

为了解决安全性问题，**HTTPS** 应运而生。简单来说，HTTPS 就是在 HTTP 和底层的 TCP 协议之间，加入了一层 **SSL/TLS 加密层**。

它们两者的核心区别，我认为主要体现在三个维度：

1. **安全性与防篡改**：HTTP 是明文裸奔，而 HTTPS 会对传输的数据进行加密，并且具有完整性校验和身份验证机制，能有效防止中间人攻击和数据篡改。
2. **默认端口**：HTTP 默认走的是 80 端口，而 HTTPS 默认使用的是 443 端口。
3. **连接开销与握手过程**：HTTP 的连接相对简单，只需要底层的 TCP 三次握手就能直接发送报文。而 HTTPS 在 TCP 握手之后，还必须进行 **TLS 握手**（用来验证证书、协商加密算法和生成对称密钥），所以在首次建立连接时，HTTPS 的耗时会比 HTTP 略长一点。但结合它带来的极高安全性，这种性能损耗是完全可以接受的，而且在现代 HTTP/2 的多路复用下，性能已经得到了极大的优化。

### 区别总结

| **对比维度**        | **HTTP (HyperText Transfer Protocol)** | **HTTPS (HTTP + Secure/TLS)**                      |
| ------------------- | -------------------------------------- | -------------------------------------------------- |
| **传输形式**        | **明文传输**（极不安全，易被抓包看光） | **密文传输**（SSL/TLS 加密，防止窃听）             |
| **默认端口**        | **80** 端口                            | **443** 端口                                       |
| **连接过程**        | 较快（仅需 **TCP 三次握手**）          | 较慢（TCP 三次握手 + **TLS 握手协商密钥**）        |
| **身份验证 (证书)** | 不需要                                 | 必须向 CA 机构申请**数字证书**，用于身份防伪       |
| **资源消耗**        | 极低（直接封包发送）                   | 较高（握手阶段和加解密阶段会消耗 CPU 资源）        |
| **SEO 权重**        | 普通                                   | **高**（Google 等搜索引擎明确优先收录 HTTPS 站点） |



## 2、如何理解UDP 和 TCP? 区别? 应用场景?

首先UDP和TCP都是**传输层**的核心协议，两者的不同在于，一个追求的是效率，一个追求的是可靠。

首先是UDP，全称是**User Datagram Protocol**，即用户数据包协议，是一个简单的**面向数据包的通信协议**，即对应用层交下来的报文，不合并，不拆分，只是在其上面加上首部后就交给了下面的网络层。

**UDP特点如下**：

- UDP 不提供复杂的控制机制，利用 IP 提供面向无连接的通信服务
- 传输途中出现丢包，UDP 也不负责重发
- 当包的到达顺序出现乱序时，UDP没有纠正的功能。
- 并且它是将应用程序发来的数据在收到的那一刻，立即按照原样发送到网络上的一种机制。即使是出现网络拥堵的情况，UDP 也无法进行流量控制等避免网络拥塞行为。

而TCP，全称是**Transmission Control Protocol**，即传输控制协议，是一种可靠、**面向字节流的通信协议**，它必须通过'三次握手'建立连接。它内部有极其复杂的确认机制、重传机制和拥塞控制，确保数据包按顺序、无差错地到达。

**TCP特点如下**：

- TCP充分地实现了数据传输时各种控制功能，可以进行丢包时的重发控制，还可以对次序乱掉的分包进行顺序控制。而这些在 UDP 中都没有。
- 此外，TCP 作为一种面向有连接的协议，只有在确认通信对端存在时才会发送数据，从而可以控制通信流量的浪费。
- 根据 TCP 的这些机制，在 IP 这种无连接的网络上也能够实现高可靠性的通信（ 主要通过检验和、序列号、确认应答、重发控制、连接管理以及窗口控制等机制实现）

| **核心对比维度**   | **TCP (Transmission Control Protocol)**        | **UDP (User Datagram Protocol)**             |
| ------------------ | ---------------------------------------------- | -------------------------------------------- |
| **连接状态**       | **面向连接**（必须先三次握手）                 | **无连接**（直接发送，无需握手）             |
| **可靠性**         | **极其可靠**（保证数据无差错、不丢失、不重复） | **不可靠**（尽最大努力交付，可能丢包或乱序） |
| **传输方式**       | **面向字节流**（存在粘包问题）                 | **面向报文**（保留边界，无粘包问题）         |
| **传输效率**       | **慢**（首部开销大，有确认机制和拥塞控制）     | **极快**（首部仅 8 字节，无各种控制机制）    |
| **双工/多播**      | 仅支持**一对一**（点对点双工通信）             | 支持一对一、**一对多、多对多**（广播/组播）  |
| **经典应用层协议** | HTTP, HTTPS, FTP, SMTP (网页、邮件、文件)      | DNS, DHCP, WebRTC (域名解析、实时音视频)     |
| **项目实战映射**   | 上传学术论文、下载模型权重文件                 | 无人机实时视频图传、模型推理画面的实时推流   |



## 3、如何理解OSI七层模型

OSI全称是：**Open System Interconnect**，即为开放式通信系统互连参考模型。OSI将计算机网络体系结构分为七层，每一层实现各自的功能和协议，并完成与相邻层的接口通信。

| **层次 (自上而下)**       | **核心职责**                          | **数据单元 (PDU)** | **经典协议 / 硬件设备**     |
| ------------------------- | ------------------------------------- | ------------------ | --------------------------- |
| 7. 应用层 (Application)   | 提供网络接口，直接面向具体业务逻辑。  | 数据 (Data)        | HTTP, HTTPS, FTP, DNS       |
| 6. 表示层 (Presentation)  | 数据格式化、压缩以及加密解密。        | 数据 (Data)        | JPEG, JSON, SSL/TLS         |
| 5. 会话层 (Session)       | 建立、管理和维护两个应用程序的对话。  | 数据 (Data)        | RPC, NetBIOS                |
| 4. 传输层 (Transport)     | 端到端的可靠/不可靠传输，按端口分发。 | 报文段 (Segment)   | **TCP**, **UDP**            |
| 3. 网络层 (Network)       | 逻辑寻址 (IP) 和路由选择。            | 数据包 (Packet)    | IP, ICMP / 路由器           |
| 2. 数据链路层 (Data Link) | 局域网内的物理寻址，差错校验。        | 帧 (Frame)         | MAC, Ethernet / 交换机      |
| 1. 物理层 (Physical)      | 传输最底层的比特流 (0和1)。           | 比特 (Bit)         | 802.11 (Wi-Fi), 网线 / 光纤 |



## 4、DNS协议是什么？说说DNS完整的查询过程？

"DNS（Domain Name System，域名系统）是互联网的一项核心应用层协议。它的主要职责是将人类可读的域名解析为网络层进行路由寻址所需的 IP 地址。DNS 默认基于 **UDP 协议**，运行在 **53 端口**。

关于 DNS 的完整查询过程，从工程实现的角度来看，可以划分为**本地缓存查找**和**网络层级查询**两个大阶段。

**第一阶段是本地缓存查找。** 当客户端发起网络请求时，系统会优先遵循就近原则查找缓存。查找顺序依次为：**浏览器缓存 -> 操作系统缓存（如 hosts 文件） -> 路由器缓存**。如果命中任意一层缓存，解析过程直接结束，返回 IP。

**第二阶段是网络层级查询。** 如果本地缓存全部未命中，客户端会向配置的**本地 DNS 服务器（LDNS）**发起解析请求。这个请求是**递归查询**的，意味着客户端只等待最终结果。

随后，LDNS 会在互联网上发起**迭代查询**，过程如下：

1. LDNS 首先向**根域名服务器（Root DNS）**发起请求，根服务器不直接返回 IP，而是返回负责对应顶级域名（如 `.com`）的**顶级域名服务器（TLD）**的地址。
2. LDNS 转向该**顶级域名服务器（TLD）**发起请求，TLD 同样不返回最终 IP，而是返回该域名注册时绑定的**权威域名服务器（Authoritative DNS）**的地址。
3. 最后，LDNS 向**权威域名服务器**发起请求，权威服务器查询自身的区域资源记录（如 A 记录），并将最终的真实 IP 地址返回给 LDNS。

拿到最终结果后，LDNS 会将这个 IP 响应给客户端，并且自身也会根据域名的 TTL（生存时间）值**将该结果缓存下来**，以加速后续相同的解析请求。客户端获取 IP 后，即可进入 TCP 建立连接的流程。"

### 查询层级及顺序

| **架构层级 / 角色**        | **查询模式**               | **核心职责与动作**                                     | **记录的数据类型**       |
| -------------------------- | -------------------------- | ------------------------------------------------------ | ------------------------ |
| **本地缓存 (Browser/OS)**  | 本机自查                   | 优先拦截请求，返回已知 IP。                            | 临时缓存数据             |
| **本地 DNS 服务器 (LDNS)** | 接收**递归**，发起**迭代** | 代理客户端完成整个公网的寻址跑腿工作，并缓存结果。     | 临时缓存数据             |
| **根域名服务器 (Root)**    | 响应迭代请求               | 全球解析起点，指引向对应的顶级域（如 `.com`, `.cn`）。 | 顶级服务器的 NS 记录     |
| **顶级域名服务器 (TLD)**   | 响应迭代请求               | 维护当前后缀的域名管理权，指引向具体的权威服务器。     | 权威服务器的 NS 记录     |
| **权威域名服务器**         | 响应迭代请求               | **最终的解析者，直接返回域名对应的真实 IP 地址**。     | 真实的 A 记录 / CNAME 等 |



## 5、说说HTTP 常见的状态码有哪些，适用场景？

"HTTP 状态码主要用于表示服务器对客户端请求的处理结果。在 RESTful API 设计和日常的网络协议调试中，状态码被严格划分为五个大类（1xx 到 5xx）。以下是日常开发中最常见、最具代表性的状态码及其适用场景：

**1xx：消息**

代表请求已被接受，需要继续处理。这类响应是临时响应，只包含状态行和某些可选的响应头信息，并以空行结束

常见的有：

- **100（客户端继续发送请求，这是临时响应）**：这个临时响应是用来通知客户端它的部分请求已经被服务器接收，且仍未被拒绝。客户端应当继续发送请求的剩余部分，或者如果请求已经完成，忽略这个响应。服务器必须在请求完成后向客户端发送一个最终响应
- **101**：服务器根据客户端的请求切换协议，主要用于websocket或http2升级

**2xx：成功响应（Success）**

- **200 OK**：最常见的成功状态。适用于标准的 GET 请求获取数据，或者 PUT 请求更新数据成功。
- **201 Created**：表示请求已成功，并因此创建了一个新的资源。通常用于 POST 请求（例如新增一条用户记录或提交表单）。
- **204 No Content**：服务器成功处理了请求，但不需要返回任何实体内容。常见于 DELETE 请求删除资源成功，或者跨域请求（CORS）的 OPTIONS 预检请求响应。

**3xx：重定向（Redirection）**

- **301 Moved Permanently**：永久重定向。表示请求的资源已经被永久分配了新的 URI。适用于网站域名更换或 HTTP 强制跳转 HTTPS。
- **302 Found**：临时重定向。表示资源临时被移动。适用于未登录用户访问受保护页面时，临时跳转到登录页。
- **304 Not Modified**：资源未修改。这是一个极其重要的缓存状态码。表示客户端发送了一个带条件（如 `If-None-Match`）的 GET 请求，服务器发现资源自上次请求以来并未改变，允许客户端直接使用本地的浏览器缓存。

**4xx：客户端错误（Client Error）**

- **400 Bad Request**：前端发送的请求存在语法错误或参数缺失，服务器无法理解。
- **401 Unauthorized**：缺乏目标资源的有效身份验证凭证。通常是前端未携带 Token，或者 Token 已过期。
- **403 Forbidden**：服务器理解请求，但拒绝执行。通常是因为用户身份已确认（有 Token），但权限等级不够（如普通用户尝试访问管理员接口）。
- **404 Not Found**：服务器上找不到请求的资源。通常是前端请求的 URL 拼写错误，或者接口已被后端下线。
- **405 Method Not Allowed**：请求方法不被允许。例如后端接口只接受 POST，前端却用 GET 发起请求。

**5xx：服务器端错误（Server Error）**

- **500 Internal Server Error**：后端代码执行抛出异常（如空指针、数据库连接失败等），导致服务器无法完成请求。
- **502 Bad Gateway**：网关错误。通常是前端请求到达了 Nginx 代理服务器，但 Nginx 转发给后端的真实服务（如 Node.js 或 Java 进程）时，发现真实服务挂掉了或无响应。
- **504 Gateway Timeout**：网关超时。Nginx 成功转发了请求，但后端服务处理时间过长（如死循环或慢查询），导致 Nginx 等待超时。

| **状态码类别**       | **具体状态码** | **英文释义**              | **核心含义**                               | **典型业务适用场景**                                        |
| -------------------- | -------------- | ------------------------- | ------------------------------------------ | ----------------------------------------------------------- |
| **1xx (信息提示)**   | `100`          | Continue                  | 服务器已收到请求头，请继续发送请求体。     | 客户端发送带有大体积 payload 的 POST 请求前的预检。         |
| **2xx (成功)**       | `200`          | OK                        | 请求成功，并返回目标数据。                 | 绝大多数成功的 GET（获取）或 PUT（更新）请求。              |
|                      | `201`          | Created                   | 请求成功，并在服务器创建了新资源。         | 成功的 POST 请求（如注册用户、发布文章）。                  |
|                      | `204`          | No Content                | 请求成功，但响应报文不包含实体数据。       | 成功的 DELETE（删除）请求；CORS 的 OPTIONS 预检请求。       |
| **3xx (重定向)**     | `301`          | Moved Permanently         | 资源已**永久**移动到新位置。               | 网站域名更换；HTTP 强制跳转 HTTPS（利于 SEO 权重转移）。    |
|                      | `302`          | Found / Moved Temporarily | 资源**临时**移动到新位置。                 | 未登录用户尝试访问受保护页面，临时跳转至登录页。            |
|                      | `304`          | Not Modified              | 资源未被修改，指示客户端使用本地缓存。     | 协商缓存生效（响应速度极快，性能优化的核心体现）。          |
| **4xx (客户端错误)** | `400`          | Bad Request               | 服务器无法理解该请求。                     | 前端传递的 JSON 格式错误；参数类型错误或缺失必填项。        |
|                      | `401`          | Unauthorized              | 缺乏目标资源的有效身份验证凭证。           | 未携带 Authorization Token；或 Token 已过期。               |
|                      | `403`          | Forbidden                 | 服务器理解请求，但拒绝授权执行。           | 跨权操作（如普通角色尝试调用 Admin 接口）。                 |
|                      | `404`          | Not Found                 | 服务器找不到请求的资源。                   | 前端请求的 URL 地址拼写错误；或请求了已下线的接口。         |
|                      | `405`          | Method Not Allowed        | 请求的方法不被目标资源支持。               | 接口要求使用 POST，但前端错误地使用了 GET 发起请求。        |
| **5xx (服务器错误)** | `500`          | Internal Server Error     | 服务器内部代码执行遇到不可预期的错误。     | 后端代码抛出异常（如 NullPointerException、SQL 语法错误）。 |
|                      | `502`          | Bad Gateway               | 网关或代理服务器收到上游服务器的无效响应。 | Nginx 运行正常，但代理转发的后端真实服务进程挂掉。          |
|                      | `504`          | Gateway Timeout           | 网关或代理服务器等待上游服务器响应超时。   | 后端服务死循环；数据库慢查询导致接口长时间无返回。          |



## 6、Axios

### 基本使用

#### 安装

```bash
# 项目中安装
npm install axios -S
# cdn 引入
```

```html
<script src="https://unpkg.com/axios/dist/axios.min.js"></script>
```

#### 导入

```js
import axios from 'axios'
```

#### 发送请求

```js
axios({
  url:'xxx',    // 设置请求的地址
  method:"GET", // 设置请求方法
  params:{      // get请求使用params进行参数拼接,如果是post请求用data
    type: '',
    page: 1
  }
}).then(res => {
  // res为后端返回的数据
  console.log(res);
})
```

### 为什么要封装

`axios` 的 API 很友好，你完全可以很轻松地在项目中直接使用。

不过随着项目规模增大，如果每发起一次`HTTP`请求，就要把这些比如设置超时时间、设置请求头、根据项目环境判断使用哪个请求地址、错误处理等等操作，都需要写一遍

这种重复劳动不仅浪费时间，而且让代码变得冗余不堪，难以维护。为了提高我们的代码质量，我们应该在项目中二次封装一下 `axios` 再使用

### 封装示例

```js
import axios from 'axios'
// 这里以 Element Plus 为例进行全局错误提示，你可以根据实际项目替换为 vant、antd 等
import { ElMessage } from 'element-plus'
// 引入路由，用于在 Token 过期时强制跳转回登录页
import router from '@/router'

// ==========================================
// 第一步：打地基 —— 创建专属的 Axios 实例
// ==========================================
const service = axios.create({
  // 基础 URL：如果你的接口都是 http://api.xxx.com/v1/users，这里就填前面的公共部分
  // 真实开发中，通常配合 Vite 的环境变量使用：import.meta.env.VITE_APP_BASE_API
  baseURL: 'http://api.company.com/v1',

  // 超时时间：10秒。如果后端 10 秒还没把数据传回来，直接打断，防止页面死等卡死
  timeout: 10000,

  // 默认请求头：告诉后端我们传过去的数据是 JSON 格式（绝大多数现代后端的标准）
  headers: {
    'Content-Type': 'application/json;charset=utf-8'
  }
})

// ==========================================
// 第二步：请求拦截器 —— "出站安检员"
// 作用：在请求发出去之前，统一加上 Token 等公共参数
// ==========================================
service.interceptors.request.use(
  (config) => {
    // 1. 从浏览器的本地缓存 (或 Pinia/Vuex) 中拿到用户的登录凭证 Token
    const token = localStorage.getItem('token')

    // 2. 如果存在 Token，就强制塞进每一次请求的 Header 里
    if (token) {
      // 注意：'Authorization' 和 'Bearer ' 是行业规范，具体要看你们后端的接口文档怎么定义的
      config.headers['Authorization'] = `Bearer ${token}`
    }

    // 3. 放行！把配置好的包裹交还给 axios，让它发出去
    return config
  },
  (error) => {
    // 请求发送本身出了致命错误时触发（极少遇到）
    console.error('请求发送异常:', error)
    return Promise.reject(error)
  }
)

// ==========================================
// 第三步：响应拦截器 —— "进站安检员"（最核心的灵魂）
// 作用：在数据回到 Vue 组件之前，剥离外壳、拦截报错
// ==========================================
service.interceptors.response.use(
  (response) => {
    // 1. 剥离网络协议外壳，直接提取后端业务数据
    // response.data 才是后端真正通过 res.send() 传过来的那个 JSON 对象
    const res = response.data

    // 2. 判断业务状态码 (假设公司约定 code: 200 为成功)
    // 注意：有些公司成功是 code: 0，这里一定要和后端对齐！！！
    if (res.code === 200) {
      // 业务完全成功，直接把核心数据 return 给 Vue 页面
      return res.data
    } else {
      // 业务失败（如：密码错误、余额不足等）
      // 统一在这里弹出错误提示，页面里就不用再写弹窗了
      ElMessage.error(res.msg || '系统业务处理异常')

      // 返回一个被拒绝的 Promise，打断流程，防止错误数据进入页面的 .then()
      return Promise.reject(new Error(res.msg || 'Error'))
    }
  },
  (error) => {
    // 3. 处理 HTTP 状态码层面的致命网络错误 (如 404, 500)
    // 保护机制：如果连 error.response 都没有，说明请求都没到后端（比如断网了）
    const status = error.response ? error.response.status : null

    switch (status) {
      case 401:
        ElMessage.error('登录凭证已过期，请重新登录')
        // 自动清理过期的 Token
        localStorage.removeItem('token')
        // 自动把用户踢回登录页面
        router.push('/login')
        break
      case 403:
        ElMessage.error('您没有权限访问该资源，拒绝访问')
        break
      case 404:
        ElMessage.error('您请求的接口路径不存在 (404)')
        break
      case 500:
        ElMessage.error('后端服务器崩溃了 (500)，请联系后端开发')
        break
      default:
        ElMessage.error(error.message || '网络连接异常，请检查网络')
    }

    return Promise.reject(error)
  }
)

// ==========================================
// 第四步：封装 RESTful 快捷方法 —— "对外取件窗口"
// ==========================================
export const http = {
  /**
   * get 请求 (查)
   * @param {String} url 请求地址
   * @param {Object} params URL问号后面的参数 (如 ?page=1)
   * @param {Object} config 针对当前接口的特殊配置 (可选)
   */
  get(url, params, config = {}) {
    return service.get(url, { params, ...config })
  },

  /**
   * post 请求 (增)
   * @param {String} url 请求地址
   * @param {Object} data 放在请求体 body 里的数据
   * @param {Object} config 特殊配置 (可选)
   */
  post(url, data, config = {}) {
    return service.post(url, data, config)
  },

  /**
   * put 请求 (改 - 全量替换)
   */
  put(url, data, config = {}) {
    return service.put(url, data, config)
  },

  /**
   * patch 请求 (改 - 局部更新)
   */
  patch(url, data, config = {}) {
    return service.patch(url, data, config)
  },

  /**
   * delete 请求 (删)
   */
  delete(url, params, config = {}) {
    // 注意：axios 原生的 delete 传参有点特殊，参数必须包在 config 的 params 属性里
    return service.delete(url, { params, ...config })
  }
}

// 默认导出 axios 实例，以备极个别极端场景需要用到原生 service
export default service
```

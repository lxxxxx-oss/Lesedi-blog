# 为什么要手写自定义底部导航栏

首先，关于为什么要舍弃微信官方极其简单的 `app.json` 原生配置，而去大费周章地手写自定义 TabBar。这完全是出于真实的业务痛点考量。在"指尖校园"这个项目中，我需要服务学生、班委、辅导员和宿管这四类完全不同的权限角色。如果使用原生的静态配置，导航栏的菜单项在代码打包时就被写死了，这意味着我无法做到"辅导员登录时看到的是审批和管理菜单，而学生登录时看到的是打卡和请假菜单"。为了实现这种严格的视图隔离，我必须脱离静态配置，手写自定义底部导航栏组件。具体来说，我在小程序根目录下创建了 `custom-tab-bar` 文件夹，在组件的初始化生命周期中，去读取用户登录后保存在本地缓存或全局状态中的角色标识，然后动态计算并渲染出一套专属于该角色的菜单列表。

## 如何利用经纬度获取、地图 SDK 计算来实现严格的地理位置签到，以及如何防止学生"代打卡"或"虚拟定位"作弊

第一个阶段，是发起人也就是老师端的"动态锚点"采集。当老师在小程序端点击"发起签到"时，我的前端代码会立刻调用微信原生的 `wx.getLocation` 接口。为了保证极高的定位精度，我会在参数中严格指定 `type` 为 `gcj02`，也就是国测局的加密坐标系，而不是普通的 WGS84。获取到老师当前的精确经纬度后，前端会将这个坐标作为本次签到任务的"动态中心锚点"，连同签到有效时间一起提交给后端数据库生成一个签到任务。

第二个阶段，是学生端的定位获取与预处理。当学生在自己手机上打开这个签到任务准备打卡时，前端同样会进行严格的权限校验。这里有个开发细节，微信要求必须在 `app.json` 中配置 `requiredPrivateInfos` 并在 `permission` 字段写明定位用途，否则接口会直接报错。获取到学生的 `gcj02` 坐标后，就进入了最关键的距离测算环节。

第三个阶段，就是核心的距离比对逻辑。为了保证计算的准确性，我集成了腾讯位置服务 SDK。在拿到老师的基准经纬度和学生当前的经纬度后，我会利用 SDK 中封装好的距离计算能力。它的底层其实是运用了复杂的 Haversine 球面距离公式，将两个地球球面坐标系的数据，转化为精确的直线物理距离（单位为米）。我的前端代码会判断这个计算出来的距离：如果两者的距离小于等于 50 米，说明学生确确实实就在老师的眼皮底下，前端就会放行，将打卡成功的状态和坐标一并提交给后端记录；如果距离大于 50 米，前端就会直接拦截本次操作，并弹窗提示"您当前不在签到范围内"。

通过这种"老师动态生成锚点 + 腾讯地图 SDK 实时球面测距 + 50 米阈值拦截"的技术链路，我们极其优雅且低成本地解决了高校里的代打卡痛点。因为哪怕学生用了虚拟定位软件，只要他不知道老师此刻站在哪个极其精确的坐标点上，他就无法把自己的虚拟坐标设置进那个 50 米的圆圈里，从而彻底杜绝了人在寝室却能帮全班签到的作弊行为。

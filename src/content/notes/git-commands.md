# Git 命令速查

## 分支工作流程

- 在自己的分支上编写代码：`git checkout taskAudit`
- 写完后，暂存（在 VS Code 点击 `+` 号暂存，一些文件不希望上传到仓库中）
- 然后提交（Commit）`--no-verify`
- 切换到主分支：`git checkout main`
- 拉取主分支最新的代码：`git pull origin main`
- 再切回到自己的分支：`git checkout taskAudit`
- 把主分支的代码合并到自己的分支上面：`git merge main`。如果有冲突，就在自己的分支上面解决
- 确认没冲突过后，再切换主分支：`git checkout main`
- 最后用主分支合并自己的分支：`git merge taskAudit`

## 基础提交

| Git 命令 | 作用与场景 |
| --- | --- |
| `git add .` | 把当前目录下所有修改过的文件放入"暂存区"（装箱）。 |
| `git commit -m "描述"` | 正常提交代码（封箱）。此时会被 Husky 和代码规范工具拦截检查。 |
| `git commit -m "描述" --no-verify` | **强行跳过检查**直接提交。（紧急情况/历史遗留报错太多时使用）。 |
| `git push origin <分支名>` | 将本地已提交的代码推送到远程服务器。 |

## 分支切换和同步（合并前必做）

| Git 命令 | 作用与场景 |
| --- | --- |
| `git status` | **最重要的命令！** 随时查看当前在哪个分支、有没有未提交的文件、是否处于冲突状态。 |
| `git checkout main` | 切换到 `main` 分支（其他分支同理）。 |
| `git pull origin main` | 从远程拉取 `main` 分支的最新代码到本地。（合并前**必须**执行，防止代码覆盖）。 |
| `git merge <分支名>` | 最普通的合并。把指定分支的代码全部合并到当前分支。 |

## 高级合并技巧（挑选文件、清理记录）

| Git 命令 | 作用与场景 |
| --- | --- |
| `git merge <分支名> --no-commit` | 合并代码，但**先不自动提交**。让代码留在暂存区，方便你去 VS Code 剔除不需要的文件。（注：如果遇到 `Fast-forward` 快进合并，此参数会失效）。 |
| `git merge <分支名> --squash` | **挤压合并（推荐）**。把目标分支上的所有历史提交，压缩成一堆纯净的代码放在暂存区。**非常适合用来挑选文件，且不会把乱七八糟的历史记录带到主分支。** |
| `git push origin <分支名> --force` | **强制推送**。当你用 `reset` 修改了本地分支的历史记录后，强行覆盖服务器上的旧记录。（注意：**绝对禁止**对 `main` 分支使用，仅限个人分支清理用）。 |

## 撤销和回退

| Git 命令 | 作用与场景 |
| --- | --- |
| `git merge --abort` | **取消合并**。合并时出现大面积飘红冲突，不想手动解决，直接用这个命令取消，回到合并前的状态。 |
| `git reset --hard origin/main` | **强制复原**。不管本地有什么乱七八糟的提交或改动，强行把本地分支变成和服务器上的 `main` 一模一样。 |
| `git reset --soft origin/main` | **温柔重置（高级操作）**。把本地比服务器多出来的提交"拆开"，代码**原封不动**保留在暂存区。 |
| `git reflog` | **查看操作录像**。记录了你最近所有的 Git 动作（包括 reset、merge）。如果代码丢了，用它来找线索。 |
| `git reset --hard HEAD@{n}` | **精准时光倒流**。配合 `reflog` 使用，把状态强行恢复到 `HEAD@{n}` 记录的那一瞬间。 |

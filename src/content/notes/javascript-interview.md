# JavaScript 面试题集锦

## 1、JavaScript中的数据类型？区别？

#### 1. JavaScript 的数据类型分类

JavaScript 中的数据类型分为两大类，目前共有 **8 种**：

- **基本数据类型（Primitive Types，7种）：**
  - `Number` (数字)
  - `String` (字符串)
  - `Boolean` (布尔值)
  - `Undefined` (未定义)
  - `Null` (空值)
  - `Symbol` (ES6 引入，表示独一无二的值，常用于对象属性名防冲突)
  - `BigInt` (ES10 引入，用于表示超出安全范围的大整数)
- **引用数据类型（Reference Types，1大类）：**
  - `Object` (对象)。注意，在 JavaScript 中，像 `Array` (数组)、`Function` (函数)、`Date` (日期)、`RegExp` (正则) 等，本质上都属于 Object 的子类型。

#### 2. 两者的核心区别（面试拿分点）

它们最根本的区别在于**在内存中的存储方式不同**，这直接决定了它们在代码运行时的行为差异。

| **维度**         | **基本数据类型 (Primitive)**                                 | **引用数据类型 (Reference)**                                 |
| ---------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **存储位置**     | **栈内存 (Stack)**                                           | **堆内存 (Heap)** （同时在栈内存中保存了一个指向该堆内存的地址指针） |
| **数据大小**     | 大小固定，占用空间小，频繁使用                               | 大小不固定，占用空间大                                       |
| **可变性**       | 值是**不可变**的（对变量重新赋值本质上是在栈里替换了新值，原值如果没被引用会被回收） | 值是**可变**的（可以随时增删改对象内部的属性）               |
| **赋值操作**     | **传值 (Pass by value)**：复制的是具体的值。变量之间修改互不影响。 | **传址 (Pass by reference)**：复制的是内存地址指针。修改其中一个变量内部的属性，会影响另一个。 |
| **比较 (`===`)** | 比较的是**值本身**是否相等                                   | 比较的是**内存地址**是否相同，即使两个对象长得一模一样，地址不同也不相等 |

## 6、== （宽松等于）和 ===（严格等于）区别，分别在什么情况使用？

"面试官您好。`===`（严格相等）和 `==`（宽松相等）的根本区别在于：**是否允许 JS 引擎在底层触发隐式类型转换（Type Coercion）**。

`===` 也就是全等操作符，它是极其冷酷的'物理校验器'。当它比较左右两边的变量时，会**首先进行物理数据类型的比对**。只要类型不同，它连值都不看，直接宣判 `false`。而 `==` 则是底层充满妥协的'抽象比较器'。当它发现两边类型不同时，非但不会报错，反而会根据一套极其复杂的《抽象相等比较算法》，强行将两边的数据降维或转换成相同的类型，然后再去比较值。

在现代前端工程规范（如 ESLint 标准）中，我们的绝对原则是：**在 99.9% 的业务场景下，无条件使用 `===`**，以彻底物理斩断隐式转换带来的不可预期 Bug。而 `==` 的生存空间已经被无限压缩，目前唯一的'合法'使用场景，仅仅是为了极其简便地同时拦截 `null` 和 `undefined`。"

| **维度**                     | **=== (严格相等 / 全等)**                                    | **== (宽松相等)**                                            |
| ---------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **底层校验规则**             | **第一步：查类型。类型不同，直接判死刑 (`false`)。** 第二步：查内存值。值相同才为 `true`。 | 发现类型不同时，立刻启动隐式转换机制。**疯狂将数据向 Number (数字) 类型降维**，直到两边类型一致再比较。 |
| **执行速度**                 | **极快**。不需要在底层调用任何类型转换函数（如 `ToNumber` 或 `ToPrimitive`）。 | **略慢**。引擎为了让不同类型变得一样，在底层要跑好几圈转换逻辑。 |
| **对 `null` 和 `undefined`** | `null === undefined` ➡️ **`false`** (因为类型不同，一个是 Null，一个是 Undefined) | `null == undefined` ➡️ **`true`** (JS 底层规范硬性规定的特例，它们俩在这个符号前是亲兄弟) |
| **经典表现对比**             | `1 === "1"` ➡️ `false` (数字 !== 字符串) `0 === false` ➡️ `false` (数字 !== 布尔) | `1 == "1"` ➡️ `true` (字符串被强转为数字 1) `0 == false` ➡️ `true` (false 被强转为数字 0) |



## 7、深拷贝浅拷贝的区别？如何实现一个深拷贝？

### 浅拷贝

在堆内存中开辟一块新空间，但**只复制对象的第一层属性**。如果属性是基本数据类型（如数字、字符串），直接复制值；如果属性是引用数据类型（如嵌套的对象、数组），它**只复制那个嵌套对象的内存地址（指针）**。修改浅拷贝出的新对象中的"基本类型属性"，原对象没事；但如果修改了"嵌套对象里的属性"，原对象会跟着一起被篡改（因为它们共用一把底层钥匙）。

#### 1、使用展开语法

**适用场景**：日常的解构赋值、Vue/React 中更新简单的组件状态。

```js
const arr = [1, 2, { a: 3 }];
const copyArr = [...arr]; 

const obj = { name: "张三", info: { age: 18 } };
const copyObj = { ...obj };
```

#### 2、`Object.assign()` 

**适用场景**：合并多个配置项。

```js
const obj = { name: "张三" };
const obj1 = { sex: "男" };
// 把 obj 的第一层属性塞进一个全新的空对象 {} 里
const copyObj = Object.assign({}, obj);

// 也可以实现将两个对象进行合并
// 把obj1的内容合并到obj里面
// obj = {name: "张三", sex: "男"}
Object.assign(obj, obj1); 
```

#### 3、数组专属：`slice()` 

原本是用来截取数组的，如果不传参数，刚好就能截取整个数组，实现浅拷贝。

```js
const arr = [1, 2, 3];
const copyArr = arr.slice();
```

#### **4、 数组专属：`concat()`** 

原本是用来把两个数组拼在一起的。如果拿一个空数组去拼，就变相实现了浅拷贝。

```js
const arr = [1, 2, 3];
const copyArr = [].concat(arr);
```



### 深拷贝

不仅开辟新空间，还会利用**递归**，将原对象内部所有层级的嵌套结构，全部在堆内存中重新克隆出一份1:1的全新实体。

#### 1.JSON.parse(JSON.stringify())

先转成字符串，再转回对象

**优点**：代码极简，一行搞定。

**致命缺点**：不能拷贝函数、`undefined`，遇到时间对象（`Date`）会变成字符串，遇到循环引用直接报错死机。

```js
const obj = { a: 1, b: { c: 2 } };
const deepObj = JSON.parse(JSON.stringify(obj));
```



#### 2.现代浏览器原生 API：`structuredClone()` 

这是 JavaScript 官方终于看不下去了，在现代浏览器里新增的原生深拷贝方法。

- **优点**：原生支持，完美处理循环引用，支持 `Date`、`Map`、`Set` 等复杂结构。
- **缺点**：依然不支持拷贝函数（遇到函数会报错），且在较老的浏览器（比如老版本微信内置浏览器）中可能不支持。

```js
const obj = { a: 1, date: new Date() };
const deepObj = structuredClone(obj);
```



#### 3.第三方库，lodash.cloneDeep

**优点**：完美无瑕。无论里面有什么奇葩数据类型、不论嵌套多深、不管有没有循环引用，它都能极其稳定地给你克隆出来。

```js
import { cloneDeep } from 'lodash';

const obj = { a: 1, b: { c: 2 }, func: () => {} };
// 一键完美深拷贝
const deepObj = cloneDeep(obj);
```



#### 4.通过循环递归实现

```js
/**
 * 手写深拷贝函数 (完整版)
 * @param {Object|Array} target - 需要被拷贝的目标对象
 * @param {WeakMap} hash - 核心武器：用于记录已经被克隆过的对象，解决循环引用引起的栈溢出
 */
function deepCopy(target, hash = new WeakMap()) {
  // 1. 物理拦截：触底反弹！
  // 只要发现当前的数据是基本类型（数字、字符串）或 null，说明拆到最底了，直接返回对应的值。
  if (typeof target !== 'object' || target === null) {
    return target;
  }

  // 2. 查阅"访问记录"（解决死循环的关键）：
  // 如果原对象 A 的属性指向了 B，B 的属性又指回了 A，无限递归会让浏览器内存直接爆炸。
  // 所以在克隆前查一下表：如果这个对象之前已经克隆过了，直接把当时克隆好的成品交出去，强行斩断死循环。
  if (hash.has(target)) {
    return hash.get(target); 
  }

  // 3. 开辟新空间：像浅拷贝一样，在内存里建造一个全新的空数组或空对象。
  const cloneTarget = Array.isArray(target) ? [] : {};

  // 4. 登记造册（留下记号）：
  // 刚建好新房间，立刻把它和老房间的对应关系记录进 hash 表里。
  // 这样如果后续在内部又遇到了这个老房间，引擎就知道该直接拿这个新房间去顶替了。
  hash.set(target, cloneTarget);

  // 5. 遍历原对象的所有属性
  for (let key in target) {
    
    // 6. 同样地，过滤掉原型链上的多余属性，只处理自身属性
    if (Object.prototype.hasOwnProperty.call(target, key)) {
      
      // 7. 魔法发生的地方：递归调用！
      // 我们不再像浅拷贝那样直接 cloneTarget[key] = target[key] 了。
      // 而是把 target[key] 这个值，再次扔进 deepCopy 函数的流水线里！
      // 如果它是对象，流水线会把它继续拆解；如果它是数字，会在第 1 步被弹出来并赋值。
      cloneTarget[key] = deepCopy(target[key], hash);
    }
  }

  // 8. 经过无数次的层层递归，所有属性都被 1:1 完美重建，返回最终的无菌克隆体。
  return cloneTarget;
}
```



## 8、闭包是什么？闭包的使用场景有哪些？

所谓闭包就是，一个函数对其周围状态的引用捆绑在一起，或者说是函数被引用包含，这样的组合就是闭包。

也就是说，闭包可以**让一个内层函数访问到其外层函数的作用域**。

这就引出了闭包的作用：

- 创建和保护私有变量
- 延长变量的生命周期

> [!NOTE]
>
> 一般函数的词法环境在函数返回后就被销毁，但是闭包会保存对创建时所在词法环境的引用，即便创建时所在的执行上下文被销毁，但创建时所在词法环境依然存在，以达到延长变量的生命周期的目的

一个简单的例子

```js
// 1. 我们定义一个父函数（相当于老家）
function createCounter() {
  // 这是老家里的一个局部变量
  let count = 0; 
  
  // 2. 在老家里，我们生了一个子函数（准备外出打工）
  function innerFunction() {
    // 3. 子函数偷偷把老家的 count 塞进了自己的背包里
    count++; 
    console.log("当前数字是：", count);
  }

  // 4. 把子函数送出老家，给外部去用
  return innerFunction; 
}

// 5. 执行父函数，把生出来的子函数存到 myCounter 里。
// 注意：此时 createCounter 已经执行完了！按理说，它内部的 count 变量应该被垃圾回收机制（GC）销毁了。
const myCounter = createCounter();

// 6. 见证奇迹的时刻：
myCounter(); // 输出: 当前数字是： 1
myCounter(); // 输出: 当前数字是： 2
myCounter(); // 输出: 当前数字是： 3
```

### 保护私有变量

参考上面的代码，要想使count++，除了调用myCounter()之外，没有任何办法能修改count的值。这就相当于把数据死死地保护在了函数的内部，只对外提供了一个极其安全的"操作接口"。这在开发复杂的组件时非常有用，可以防止别人瞎改你的核心数据。

### 延长变量生命周期

这是闭包最为常见的用途，顾名思义使用闭包能够让变量一直活着。

```js
// 防抖函数
function debounce(fn, delay) {
  let timer = null; // 这个就是装进背包里的变量！
  
  return function (...args) {
    if (timer) clearTimeout(timer); // 子函数在使用背包里的 timer
    timer = setTimeout(() => fn.apply(this, args), delay);
  }
}
```

如果没有闭包，每次调用防抖函数时，`timer` 都会被重新变成 `null`，防抖就彻底失效了。正是因为闭包把 `timer` 装进了背包里一直保持着状态，下一次触发时才能准确找到上一次的定时器并把它清空。

### 闭包的缺点

如果不是某些特定任务需要使用闭包，在其它函数中创建函数是不明智的，因为闭包在处理速度和内存消耗方面对脚本性能具有负面影响。如果你在代码里疯狂地制造闭包，而且闭包里装的都是特别巨大的对象（比如一大段 DOM 节点），并且用完之后忘记把子函数销毁掉，这些庞大的背包就会把浏览器的内存一点点吃光，导致页面越来越卡，最终崩溃。这就是传说中的**"内存泄漏"**。"闭包的最终销毁是由浏览器的垃圾回收器（GC）通过'标记-清除算法'自动完成的。但是，因为闭包往往被外部变量或 DOM 事件监听器强引用着，导致它处于'可达'状态，GC 无法回收。因此，我们需要手动将引用它的外部变量赋值为 `null`，或者移除对应的事件监听器，从物理层面上打断它的'可达性'。这样在 GC 下一次运行时，就会自动将其销毁，从而避免内存泄漏。"



## 9、说说对作用域链的理解

"作用域链就是：它是 JavaScript 引擎在执行代码时，用来**查找变量的一套单向、向上的底层寻址机制**。当代码在一个特定的环境中执行并试图访问某个变量时，引擎首先会在'当前函数的局部作用域'中寻找；如果找不到，它就会顺着代码在物理层面（词法层面）上的嵌套关系，一层一层地向外层作用域去查找，直到抵达最外层的'全局作用域'（如浏览器中的 `window`）。如果到了全局依然没找到，就会抛出 ReferenceError 报错。这种由内向外、如同链条一般递进的查找路径，就是作用域链。

```js
let a = 1;  // 这里是全局作用域，有一个 a 叫 1

// printA 出生在全局作用域上
function printA() {
  console.log(a); // printA 说：我要找 a！
}

// run 也出生在全局作用域"上
function run() {
  let a = 999; // run 的函数作用域有一个 a 叫 999
  
  // 此时，run 把 printA 请到了自己的家里来做客执行
  printA(); 
}

// 执行！
run(); // 1
```

它的核心物理本质在于：**作用域链是在代码书写（声明）的那一刻就已经在物理位置上死死绑定的（也就是词法作用域），它绝对不会因为你在哪里调用了这个函数而发生任何改变。**"

**作用域 (Scope) —— 就是"地盘"**

作用域就是变量能够合法活动的**地盘（物理空间）**。比如全局作用域就是"公共广场"，函数内部的作用域就是"私人家里"。在家里（函数内）可以随便用广场上（全局）的东西，但广场上的人进不来你家。

**词法作用域 (Lexical Scope) —— 就是"出生地原则"**

在 JavaScript 中，一个函数的地盘范围，是由它**在代码编辑器里写在哪一行（出生地）决定的，并且永远焊死，绝对不改变。**不管这个函数以后被拉去哪里执行，它永远只认自己的老家。这就是"词法作用域"的唯一真相：**看代码的物理书写位置，不看执行位置。**

**作用域链 (Scope Chain) —— 就是"回老家找东西的路线"**

当函数需要用一个变量时，它会先摸摸自己的口袋（当前地盘）；如果没有，它就会顺着一条路**回到自己的出生地（老家）**去找，如果老家没有，就去老家外面的广场找。这条"自己口袋 -> 老家 -> 广场"的找东西路线，也就是就叫作用域链。

## 10、JavaScript原型，原型链 ? 有什么特点？

"面试官您好。我对原型的理解是：它是 JavaScript 实现**成员共享**与**继承**的核心机制。

在底层物理层面，每一个**构造函数**（制造厂）都有一个 **`prototype` 属性**（共享工具间），而由它创建出来的每一个**对象实例**（产品），都会自带一个 **`__proto__` 指针**（门禁卡），这个指针死死地指向构造函数的 `prototype`。

当我们访问一个对象的属性或方法时，引擎会先在对象**自身属性**中查找；如果找不到，就会顺着 `__proto__` 这条线去它的**原型对象**中查找，这种层层递进的查找路径就是**原型链**。它最终会指向 **`Object.prototype`**，如果还没找到，则指向 **`null`**。

原型链最大的特点就是**内存优化**，它让数千个实例能共用同一个方法，而不需要重复创建。同时它还具有**就近原则**，即实例身上的同名属性会'遮蔽'原型上的属性。"

### 什么是原型？

在JS中，每个对象在被创建的那一刻，都会自带一个"隐藏属性（在底层叫 `[[Prototype]]`，在浏览器里通常显示为 `__proto__`）。这个隐藏属性指向了另一个对象，那个被指向的对象，就叫做它的**"原型"（Prototype）**。

想象你经营着一家**制造厂**。

- **制造厂 (构造函数)**：专门负责生产"机器人"。
- **共享工具间 (显式原型 `prototype`)**：厂房里有一个公共区域，放着所有机器人都能共用的"说明书"和"配件（方法）"。
- **门禁卡 (隐式原型 `__proto__`)**：每个机器人出厂时，胸口都挂着一张门禁卡，这张卡刷一下就能进入那个"共享工具间"。

| **专业术语**                 | **物理形态**                                 | **它的本质是什么？（大白话）**                               |
| ---------------------------- | -------------------------------------------- | ------------------------------------------------------------ |
| **`prototype`** （显式原型） | 只有**函数（尤其是构造函数**）才拥有的属性。 | **"共享工具间"**。 当一个函数被创建时，引擎会自动给它配一个大房间。这个房间就是用来存放所有实例可以共用的方法的。 |
| **`__proto__`** （隐式原型） | **每一个对象都拥有的隐藏属性。**             | **"访问工具间的门禁卡"**。 当对象被创建出来时，它的体内会被塞入一张门禁卡，这张卡死死地指向了创造它的那个构造函数的 `prototype` 工具间。 |



### 什么是原型链？

既然每一个对象都有一个属性会指向原型对象，那么原型对象也可能拥有原型，并从中继承方法和属性，一层一层、以此类推。这种关系常被称为原型链 (prototype chain)，它解释了为何一个对象会拥有定义在其他对象中的属性和方法。

当你命令一个机器人"去跳舞"时：

1. **看口袋**：机器人先看自己身上有没有装跳舞插件。
2. **刷卡进屋**：如果没有，它会刷胸口的"门禁卡（`__proto__`）"，进到"制造厂工具间（`prototype`）"里找。
3. **逐级上报**：如果工具间也没有，它会顺着工具间后门的路线图，去"总公司（`Object.prototype`）"的仓库找。
4. **无功而返**：如果连总公司也没有（直到遇见 `null`），它才会告诉你"我不会"。

也就是：

**属性查找机制 (Property Lookup)**：当你访问对象的属性（如 `obj.prop`）时，引擎执行的操作。

**自身属性检查 (OwnProperty)**：首先在对象本身的 **环境记录 (Environment Record)** 中查找。

**原型委托 (Delegation)**：如果找不到，则沿着 **`[[Prototype]]`（即 `__proto__`）** 链条向上追溯。

**原型链终点**：所有普通对象的原型链最终都会指向 **`Object.prototype`**，而 `Object.prototype.__proto__` 为 **`null`**。



#### 原型链的四大特点

| **特点名称** | **专业术语描述**       | **实际工程意义**                                             |
| ------------ | ---------------------- | ------------------------------------------------------------ |
| **内存共享** | **Method Sharing**     | 多个实例共享同一个函数引用，避免了重复创建函数导致的 **内存冗余 (Memory Overhead)**。 |
| **动态扩展** | **Dynamic Extension**  | 即使实例已创建，修改构造函数的 `prototype` 仍能让所有存量实例实时获得新方法。 |
| **属性遮蔽** | **Property Shadowing** | 如果实例拥有与原型同名的属性，查找逻辑会"就近触发"，实例属性会 **覆盖 (Shadow)** 原型属性。 |
| **单继承性** | **Single Inheritance** | 一个对象在物理上只能拥有一个直接原型链接。虽然可以形成长链，但它是单向线性的。 |



## 11、JavaScript中如何实现继承

在 JavaScript 中，继承的底层物理本质是**原型链的链接**。在 ES6 以后，我们普遍使用 **`class` 和 `extends`** 关键字来实现继承，它在语法上更接近传统面向对象语言，底层自动处理了原型的指向。但在底层机制上，它其实是 **'寄生组合式继承'** 的语法糖：即通过 **`call`** 借用父类构造函数来继承**实例属性**，通过 **`Object.create`** 链接原型对象来继承**共享方法**。这种方式既保证了每个子类实例拥有独立的属性，又实现了方法的内存级共享，是目前最严谨的物理实现方案。"



## 12、谈谈对this对象的理解

"面试官您好。我对 `this` 的理解是：它是一个**执行上下文**中的属性，其指向是在**函数被调用时（运行时）动态绑定**的，而不是在编写时确定的。它的指向完全取决于函数的**调用位置**和**调用方式**。在 JavaScript 中，判断 `this` 指向通常遵循五条核心准则：默认绑定、隐式绑定、显式绑定、`new` 绑定，以及箭头函数的词法绑定。

this并不是只有写出来才存在。它是**执行上下文**的三大组成部分之一（另外两个是变量对象和作用域链）。每当函数被调用时，引擎都会自动创建一个 **`this` 绑定（this binding）**。哪怕函数体是空的，引擎也会根据**调用方式**，计算出当前环境下的 this指向谁。所以，`this` 不仅仅是一个对象，它更是一种**运行时状态**。即使没有显式调用或显式书写，它作为函数执行的'宿主环境'依然是物理存在的。""

### 五种绑定规则

| ------------------------------------ | ------------------------------------------------------------ | ------------------------------------------------------------ | ------------------------------------------------------------ |

## 13、JavaScript中执行上下文和执行栈是什么？

"面试官您好。我对这两个概念的理解是：**执行上下文**是 JavaScript 引擎在执行代码前的**环境准备**，而**执行栈**则是管理这些环境的**物理容器**。每当引擎开始运行代码，它都会创建一个**执行上下文**，里面封装了变量对象、作用域链和 `this` 指向。

| **维度**                          | **子项**                       | **核心动作 / 职责**                        | **物理表现**                                                 |
| --------------------------------- | ------------------------------ | ------------------------------------------ | ------------------------------------------------------------ |
| 第一阶段： 创建阶段 *(Creation)*  | **内存分配 & 提升**            | 引擎扫描代码，给变量和函数开辟空间。       | **变量提升**发生在此。`var` 初始化为 `undefined`；函数声明直接存入内存；`let/const` 仅记录名称（进入 TDZ）。 |
|                                   | **确定 `this` 指向**           | 根据函数调用方式，决定 `this` 绑定的对象。 | 在执行之前，`this` 的值就已经在这个阶段被计算并填入上下文槽位了。 |
| 第二阶段： 执行阶段 *(Execution)* | **逐行扫描 & 赋值**            | 引擎开始正式跑代码，把真实的值填进变量里。 | 此时变量才真正可用。如果遇到函数调用，则开启一个新的"函数执行上下文"并入栈。 |
| ---                               | ---                            | ---                                        | ---                                                          |
| **组成要素 A**                    | **变量环境** *(Variable Env)*  | **"旧式保险箱"**：专门存放 `var` 和函数。  | 它是为了兼容老代码，支持 `var` 的**变量提升**特性。          |
| **组成要素 B**                    | **词法环境** *(Lexical Env)*   | **"现代储物柜"**：存放 `let/const`。       | 包含一个**外部引用 (Outer)**。找变量时如果自己没有，就顺着这个引用去找"老家"，这便是**作用域链**。 |
| **组成要素 C**                    | **This 绑定** *(This Binding)* | **"动态身份标签"**                         | 存储了当前环境的 `this` 引用，决定了代码里 `this` 到底指代谁。 |

调用栈（Call Stack），是 JavaScript 引擎用来管理代码执行顺序的一种**后进先出（LIFO）**的数据结构。因为 JavaScript 是**单线程**的，同一时间只能做一件事。当引擎开始解析脚本时，会首先把'全局执行上下文'压入栈底。之后，每当代码调用一个函数，引擎就会为这个函数创建一个全新的'函数执行上下文'，并把它压入栈顶。引擎永远只会执行当前**栈顶**的上下文。当栈顶的函数执行完毕后，它的上下文就会被**弹出（Pop）**销毁，控制权随之交还给它下面的上下文。如果代码中存在无限递归，不断地向栈内压入上下文，超出了引擎的最大分配空间，就会导致**栈溢出（Stack Overflow）**错误。"

### 执行栈的四大规则

**单核独裁（Single Threaded）**：整个栈只有一个主线程在干活。没有并发，没有多任务。

**同步阻塞（Synchronous）**：栈里的代码是一行一行同步执行的。如果栈顶的那个函数是个死循环，或者是个极其耗时的巨大计算，整个栈就会被**卡死（阻塞）**，页面就会处于假死状态。

**后进先出（LIFO - Last In, First Out）**：就像往一个极窄的薯片筒里塞薯片。最后塞进去的那一片（最后调用的内部函数），必须最先被吃掉（执行完毕并弹出）。

**容量有限（Finite Memory）**：这个"薯片筒"的高度是有限的（通常是 1 万步调用左右）。塞满了就会炸（栈溢出报错）。



## 14、说说JavaScript中的事件模型 

"面试官您好。JavaScript 中的事件，可以理解就是在 HTML 文档或者浏览器中发生的一种交互操作，使得网页具备互动性， 常见的有加载事件、鼠标事件、自定义事件等。JavaScript 的事件模型描述了事件在 DOM 树中传播的**物理路径**。

现代 W3C 标准将事件传播分为三个阶段：**捕获阶段**（从根部向目标下行）、**目标阶段**（触发目标节点）和**冒泡阶段**（从目标向根部回溯）。

默认情况下，绝大多数事件处理都在**冒泡阶段**触发。理解这一模型的最大工程价值在于**'事件委托（Event Delegation）'**，即通过在父级节点监听事件，利用冒泡机制管理成百上千个子节点的交互。这不仅能极大减少内存中'事件监听器'的数量，还能自动处理动态增删子节点的交互逻辑，是前端性能优化的核心手段之一。"

| **传播阶段**                  | **物理路径 (大白话)**                                        | **传播方向**        | **核心特征**                                               |
| ----------------------------- | ------------------------------------------------------------ | ------------------- | ---------------------------------------------------------- |
| **1. 捕获阶段** *(Capturing)* | **由表及里**：信件从 `window` 出发，经过 `html`、`body` 一路向下，直到找到目标。 | `window` ➡️ `target` | 很少在此时执行逻辑，主要用于拦截或预处理。                 |
| **2. 目标阶段** *(Target)*    | **正中靶心**：信件送达目标元素（比如被点击的按钮）。         | 到达 `target`       | 事件在该元素身上触发绑定的函数。                           |
| **3. 冒泡阶段** *(Bubbling)*  | **由里及表**：信件从目标出发，原路返回 `window`，沿途通知所有父级。 | `target` ➡️ `window` | **最常用的阶段**。绝大多数交互（如点击、滑动）都在此处理。 |



## 15、typeof 与 instanceof 的区别 

"面试官您好。`typeof` 和 `instanceof` 都是 JavaScript 中用来判断数据类型的操作符，但它们的底层机制和适用场景截然不同。

**`typeof`** 主要用于判断**基础数据类型**。它返回一个表示类型的字符串。它的底层机制是根据变量的机器码低位标识来判断的。它的缺点是无法精准区分引用类型（除了函数），且存在 `typeof null === 'object'` 这种历史遗留 Bug。

**`instanceof`** 主要用于判断**引用数据类型**（对象）。它返回一个布尔值。它的底层机制是沿着对象的**原型链（`__proto__`）**向上查找，看能否找到对应构造函数的 `prototype` 属性。它的缺点是无法判断基础数据类型，且在跨 iframe 环境下会失效。

在实际工程中，为了实现最严谨的类型判断，我通常会封装 `Object.prototype.toString.call()` 方法，它可以精准返回任何变量的绝对类型。"

| **对比维度** | **typeof **                                                  | **instanceof**                                               |
| ------------ | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **测试目标** | 变量本身                                                     | 对象与构造函数的关系                                         |
| **返回值**   | 小写字符串（如 `"string"`, `"object"`）                      | 布尔值（`true` 或 `false`）                                  |
| **底层机制** | 检查底层机器码标志位                                         | 顺着隐式原型链 (`__proto__`) 向上比对                        |
| **擅长领域** | **基础数据类型** (`string`, `number`, `boolean`, `undefined` 等) | **引用数据类型** (`Array`, `RegExp`, `Date` 及自定义类)      |
| **致命盲区** | 无法区分普通对象、数组、日期等；`null` 会误判为 `"object"`。 | 无法检测基础数据类型；在存在多个全局环境（如跨 iframe）时会失效。 |

### 万能的检测法

如果需要通用检测数据类型，可以采用`Object.prototype.toString`，调用该方法，统一返回格式`"[object Xxx]"` 的字符串

```js
Object.prototype.toString({})       // "[object Object]"
Object.prototype.toString.call({})  // 同上结果，加上call也ok
Object.prototype.toString.call(1)    // "[object Number]"
Object.prototype.toString.call('1')  // "[object String]"
Object.prototype.toString.call(true)  // "[object Boolean]"
Object.prototype.toString.call(function(){})  // "[object Function]"
Object.prototype.toString.call(null)   //"[object Null]"
Object.prototype.toString.call(undefined) //"[object Undefined]"
Object.prototype.toString.call(/123/g)    //"[object RegExp]"
Object.prototype.toString.call(new Date()) //"[object Date]"
Object.prototype.toString.call([])       //"[object Array]"
Object.prototype.toString.call(document)  //"[object HTMLDocument]"
Object.prototype.toString.call(window)   //"[object Window]"
```

### 手写一个万能的类型检测

```js
function getType(obj) {
  let type = typeof obj;
  
  // 1. 基础类型和 function，走高速通道直接返回（全小写）
  if (type !== "object") {
    return type;
  }
  
  // 2. 复杂对象和 null，走精准通道，并通过 toLowerCase 统一格式
  return Object.prototype.toString
    .call(obj)
    .replace(/^\[object (\S+)\]$/, '$1')
    .toLowerCase(); 
}

// 测试结果：
// getType(123)   -> "number"
// getType([])    -> "array"
// getType(null)  -> "null"
```



## 16、说说new操作符具体干了什么？

"面试官您好。在 JavaScript 中，`new` 操作符的作用是**通过构造函数创建一个实例对象。**

当我们使用 `new` 调用一个函数时，V8 引擎在底层偷偷替我们完成了**四个物理步骤**： 

| **步骤**   | **底层动作**                            | **物理比喻 (大白话)**                                        |
| ---------- | --------------------------------------- | ------------------------------------------------------------ |
| **Step 1** | **创建空对象 (Create Object)**          | **"批地基"**：引擎在内存里凭空捏出一个全新的空对象 `{}`。    |
| **Step 2** | **链接原型 (Link Prototype)**           | **"发门禁卡"**：给这个空对象塞入 `__proto__`，指向制造厂的 `prototype` 工具间。 |
| **Step 3** | **绑定 `this` 并执行 (Bind & Execute)** | **"按头装修"**：把构造函数里的 `this` 全部替换成这个新对象，然后执行代码，把属性（家具）塞进新对象里。 |
| **Step 4** | **返回对象 (Return Instance)**          | **"交房"**：把装修好的新对象交给你。*(注：如果包工头自己私带了另一套建好的房子 return 出来，就按包工头的给。)* |

### 手写一个new操作符

```js
function mynew(Func, ...args) {
    // 1.创建一个新对象
    const obj = {}
    // 2.新对象原型指向构造函数原型对象
    obj.__proto__ = Func.prototype
    // 3.将构建函数的this指向新对象
    let result = Func.apply(obj, args)
    // 4.根据返回值判断
    return result instanceof Object ? result : obj
}
```



## 17、AJAX原理是什么？如何实现？ 

AJAX的全称是"**异步JavaScript和XML**"（Asynchronous JavaScript and XML）。

它的核心原理是：利用浏览器内置的**XMLHttpRequest（XHR）**对象作为桥梁，在不重新加载页面的前提下，在后台与服务器进行**异步的数据交换**。这意味着我们可以在页面不白屏、不刷新的状态下，实现局部的动态更新。

在底层实现上，一次完整的AJAX请求通常分为四个步骤：

1、实例化**XMLHttpRequest**对象

2、调用open()方法配置请求的类型、URL以及是否异步

3、设置回调函数监听**onreadystatechange**事件，以此捕获底层的状态变化

4、调用**send()**方法正式将网络请求发往服务器，并在回调中处理返回的数据。

### 用原生 JS 给我写一个 GET 请求

```js
// 1、实例化XMLHttpRequest对象
const xhr = new XMLHttpRequest();

// 2、调用open()方法配置相关参数
// 参数：请求方法 'GET', 请求地址 url, 是否异步 true (默认就是 true)
xhr.open('GET', 'https://....', true);

// 3、设置回调函数监听onreadystatechange事件，以此捕获底层的状态变化
xhr.onreadystatechange = function() {
  // readyState === 4 代表请求完成
  if (xhr.readyState === 4) {
    // status === 200 HTTP 成功状态码
    if (xhr.status === 200) {
      // responseText 就是服务端返回的字符串数据
      console.log('拿到服务器的数据了:', xhr.responseText);
      // 通常我们会把它转成 JSON 对象
      const data = JSON.parse(xhr.responseText);
    } else {
      console.error('服务器出错了，状态码:', xhr.status);
    }
  }
};

// Step 4: 服务员，出发！ (发送请求)
// 如果是 POST 请求，这里会传入请求体数据 xhr.send(JSON.stringify(body))
xhr.send(null);
```



## 18、bind、call、apply 区别？如何实现一个bind?

bind、call、apply都是**函数原型**上面的原生方法，它们的共同点都是**改变函数执行时的上下文，简单来说就是用来显式的改变函数内部this的指向**

它们的核心区别是：**执行时机**和**传参方式**

- **`call` 和 `apply` 是同步立刻执行的**。它们的区别仅仅在于传参格式：`call` 接收的是逗号分隔的参数列表，而 `apply` 接收的是一个数组。
- **`bind` 是延迟执行的**。它不会立刻调用函数，而是利用闭包的特性，返回一个绑定了指定 `this` 和初始参数的**新函数**，供以后在特定时机（如事件点击、定时器）调用。

| **方法**    | **核心作用 / 经典实战场景**                                  | **执行时机** | **传参形式 (首字母记忆法)**                  | **返回值**                  |
| ----------- | ------------------------------------------------------------ | ------------ | -------------------------------------------- | --------------------------- |
| **`call`**  | **继承父类**。 当对象 A 想用对象 B 的方法，但自己身上没有时，直接"强行借用"并立即执行。 • *实战 1*：`Object.prototype.toString.call(obj)` 借用万能测型器。 • *实战 2*：在子类构造函数中借用父类 `Parent.call(this, name)` 实现属性继承。 | **立即执行** | **C**omma (逗号分隔)： `(obj, arg1, arg2)`   | 原函数的**运行结果**        |
| **`apply`** | **化零为整** 。 作用和 `call` 完全一样，但它能把一个**数组**自动拆解成一个个独立的参数传给函数。 • *实战 1*：在 ES6 扩展运算符 `...` 出来之前，用来求数组最大值 `Math.max.apply(null, [1, 5, 3])`。 • *实战 2*：合并两个数组 `arr1.push.apply(arr1, arr2)`。 | **立即执行** | **A**rray (数组包裹)： `(obj, [arg1, arg2])` | 原函数的**运行结果**        |
| **`bind`**  | **保存上下文 / 函数柯里化**。 它不想立刻执行，而是要**预先锁定**好函数的 `this` 和部分参数，生成一个"代理人"留着以后慢慢用。 • *实战 1*：给 `setTimeout` 里的回调函数锁定 `this`。 • *实战 2*：React/Vue 的类组件中绑定事件处理器。 • *实战 3*：偏函数/柯里化（先传一部分参数，剩下的以后传）。 | **延迟执行** | **C**omma (逗号分隔)： `(obj, arg1, arg2)`   | **一个全新的函数** (代理人) |



### 实现一个bind

```js
// 这里的 myBind 是挂在所有函数的"祖宗原型"上的。
// 所以你可以这样用：任意函数.myBind(目标对象)
Function.prototype.myBind = function (context) {
    
    // ==========================================
    // 第一步：防呆设计（防止乱调用）
    // ==========================================
    // 这里的 this 指向的是【谁调用了 myBind】。也就是那个"原函数"。
    // 比如 Person.myBind(...)，此时 this 就是 Person。
    // 如果别人瞎搞，用一个普通对象去调 myBind，直接报错拦截。
    if (typeof this !== "function") {
        throw new TypeError("只有函数才能调用 myBind 呀！");
    }

    // ==========================================
    // 第二步：收集初始物资（闭包缓存）
    // ==========================================
    // 1. 拿参数：arguments 是 myBind 接收到的所有参数。
    // 我们用 [...arguments].slice(1) 把第一个参数（context，也就是目标老板）切掉。
    // 剩下的就是你要预先绑定的参数（比如上面的 "参数1"），存到 args 里。
    const args = [...arguments].slice(1);
          
    // 2. 存函数：这里的 this 还是那个"原函数"。
    // 我们把它存进 fn 变量里，利用【闭包】特性，永远记在这个作用域里。
    // 后面我们要替原函数干活，总得知道原函数是谁吧！
    const fn = this;

    // ==========================================
    // 第三步：制造并返回"代理人"
    // ==========================================
    // bind 的核心特点是【不立刻执行】，而是返回一个新函数（代理人）。
    // 将来别人调用的，其实是这个 Fn 函数。
    const Fn = function () {
        
        // 🌟 这是全场最烧脑的一行！我们把它拆成 3 个部分看：
        // return fn.apply( 老板是谁 , 参数是什么 );
        
        return fn.apply(
            
            // 【老板是谁？】—— 处理 new 的霸王条款
            // 此时的 this，是【将来别人调用 Fn 时】的 this。
            // 情况 A：别人直接当普通函数调用 BoundFn()。
            //        此时 this 肯定不是 Fn 造出来的实例（this instanceof Fn 为 false）。
            //        那就老老实实听 bind 的话，把老板设为你当初传进来的 context。
            // 情况 B：别人用了 new 操作符！ new BoundFn()！
            //        遇到 new，JS 引擎会在底层造一个全新的空对象，此时 this 就指向这个新对象。
            //        那么 this instanceof Fn 就会是 true！
            //        根据规范，new 的优先级最大，所以我们要无视当初传的 context，直接用这个新造出来的 this！
            this instanceof Fn ? this : context, 
            
            // 【参数是什么？】—— 参数合并（柯里化）
            // 注意！这里的 arguments 已经不是外层 myBind 的参数了！
            // 它是别人将来调用这个代理人 Fn 时，新传进来的参数（比如 "参数2"）。
            // 我们把当年存下来的老参数 args，和现在的新参数拼在一起，一起喂给原函数 fn。
            args.concat([...arguments])
        ); 
    };

    // ==========================================
    // 第四步：打通任督二脉（原型链对接）
    // ==========================================
    // 为什么要写这一步？因为如果你以后对这个代理人用了 new 操作符。
    // 比如：const obj = new Fn()。
    // obj 的 __proto__ 默认是指向 Fn.prototype 的。
    // 但 Fn 只是个空壳代理人啊！真正干活的、有方法的，是那个"原函数" fn！
    // 所以，我们要把 Fn 的原型，和 fn 的原型连起来！
    if (fn.prototype) {
        // 使用 Object.create 创建一个干净的中间层，
        // 这样即使别人修改了 Fn 的原型，也不会弄脏原函数 fn 的原型。
        Fn.prototype = Object.create(fn.prototype);
    }

    // 完工！把这个组装好的"完美代理人"交出去。
    return Fn;
}
```



## 19、对事件循环的理解

"面试官您好。我对事件循环的理解是：它是 JavaScript 引擎在**单线程**环境下，处理**异步非阻塞**任务的核心调度机制。

由于 JS 主线程同一时间只能执行一个任务，它会将任务分为**同步任务**和**异步任务**。同步任务直接在**执行栈（Call Stack）**中顺序执行；而异步任务会被挂起到宿主环境（如浏览器的 Web APIs）中执行，一旦有了结果，就会被推入**任务队列（Task Queue）**中等待，而异步任务又被分为**微任务（Microtask）和宏任务（Macrotask）**

事件循环的流程为：同步任务进入主线程，即主执行栈，异步任务进入任务队列，主线程内的任务执行完毕为空，会去任务队列读取对应的任务，推入主线程执行。上述过程的不断重复就是事件循环

| **任务分类**       | **执行优先级与时机**                                         | **常见 API (🔥 面试必背)**                                    | **物理特征与生动比喻**                                       |
| ------------------ | ------------------------------------------------------------ | ------------------------------------------------------------ | ------------------------------------------------------------ |
| 微任务 (Microtask) | **优先级极高**。 在当前宏任务执行结束后、**浏览器进行下一次 UI 渲染之前**，**一次性清空**队列里所有的微任务。 | • `Promise.then / catch / finally` • `MutationObserver` (监听 DOM 变化) • `queueMicrotask()` • `process.nextTick` *(Node.js 独有，且优先级比 Promise 还要高一点点)* | **"VIP 绿色通道"** 只要主线程一空，必须把 VIP 等候区里的所有人全部叫号办完，哪怕在办理期间又来了新的 VIP，也会跟着一起办完，绝不拖延到下一轮。 |
| 宏任务 (Macrotask) | **优先级较低**。 等微任务队列彻底清空后，事件循环才会去宏任务队列里，**每次只取出一个**宏任务来执行。 | • `setTimeout` • `setInterval` • `setImmediate` *(Node.js 独有)* • `script` (页面初始化的整体同步代码本身就是一个巨大的宏任务) • 网络请求回调 (AJAX / fetch) • 真实的 UI 交互事件回调 (点击、拖拽等) | **"普通大厅等候区"** 极其讲规矩，每次只叫一个号。办完一个普通号，大堂经理（事件循环）都会立刻回头去查一遍 VIP 区有没有人，如果有，就得优先去办 VIP。 |



## 20、DOM的常见操作有哪些？

DOM操作本质就是JavaScript对HTML节点进行**增、删、改、查**，在现代前端开发中，常用的API主要分为四大类：

| **操作大类 (CRUD)** | **现代核心 API**                         | **物理作用 (大白话)**                                    | **极简代码示例**                                             | **🔥 大厂面试避坑与性能考点**                                 |
| ------------------- | ---------------------------------------- | -------------------------------------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| 🔍 查 (查找节点)     | `querySelector` `querySelectorAll`       | 像写 CSS 一样，精准抓取页面上的 1 个或一组元素。         | `const btn = document.querySelector('#btn');`                | **避坑**：坚决弃用 `getElementsByClassName`！因为它返回的是"动态集合"，DOM 变了它跟着变，极易导致 `for` 循环死循环。`querySelectorAll` 返回的是静态的 `NodeList`，极其安全且支持 `forEach`。 |
| 🛠️ 增 (创建与挂载)   | `createElement` `append` `insertBefore`  | 凭空捏造一个标签，然后把它挂载到父节点里面。             | `const div = document.createElement('div');` `parent.append(div, '纯文本');` | **性能必考**：遇到"如何一次性插入 10000 个节点"的问题时，绝对不能用循环直接 `append`（会导致 10000 次重排卡死）。必须创建一个 **`DocumentFragment` (文档碎片)** 作为隐形推车，把节点装好后一次性上树。 |
| ✏️ 改 (修改属性)     | `textContent` `classList` `setAttribute` | 改变标签内部的文字、样式类名或自定义属性（如 data-id）。 | `el.textContent = '你好';` `el.classList.toggle('active');` `el.setAttribute('data-id', 1);` | **安全必考**：面试官问"如何防止 XSS（跨站脚本攻击）？" 答：坚决不使用 `innerHTML` 插入用户输入的内容，必须使用 `textContent`，它会把所有恶意脚本作为纯文本渲染，绝对安全。 |
| 🗑️ 删 (移除节点)     | `remove()`                               | 让元素自己把自己从 DOM 树上销毁，干脆利落。              | `badNode.remove();`                                          | **优雅度考察**：不要再写老掉牙的 `badNode.parentNode.removeChild(badNode)` 了，现代浏览器直接调用元素自身的 `remove()` 即可完成自杀。 |



## 21、说说对BOM的理解，常见的BOM对象有哪些？

BOM（Browser Object Model），浏览器对象模型，提供了独立于内容与浏览器窗口进行交互的对象。其作用就是跟浏览器做一些交互效果，比如如何进行页面的后退、前进、刷新，浏览器窗口发生变化，滚动条滚动，等等

BOM 的核心和顶级对象是 `window`，所有的全局变量和全局函数最终都会挂载到它身上。除了 `window` 本身，BOM 家族还包含了几个极其重要的子对象： 控制网页跳转和 URL 解析的 **`location`** 对象； 掌控浏览器历史记录、也是现代单页面应用（SPA）路由基石的 **`history`** 对象； 包含浏览器和设备软硬件信息的 **`navigator`** 对象； 以及获取屏幕分辨率信息的 **`screen`** 对象。

在实际的现代工程中，我用到最频繁的 BOM API 就是通过 `history.pushState` 和监听 `popstate` 事件，或者操作 `location.hash`，来实现无刷新的前端路由跳转。"

| **BOM 对象**    | **物理角色 (大白话)**                                 | **核心属性与方法 (🔥 必考)**                                  | **实战工程痛点 / 面试高频考点**                              |
| --------------- | ----------------------------------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **`window`**    | **"大管家 / 全局老总"** 代表整个浏览器窗口。          | • `innerWidth` / `innerHeight` (视口宽高) • `setTimeout` / `setInterval` • `alert()` / `confirm()` | **全局污染问题**：如果在 JS 里不用 `let/const` 声明变量，变量会直接变成 `window` 的属性，导致内存泄漏和命名冲突。 |
| **`location`**  | **"导航员"** 专门负责解析和修改当前页面的 URL。       | • `href` (完整 URL) • `search` (?后面的查询字符串) • `hash` (#后面的哈希值) • `reload()` (刷新页面) | **Hash 路由底层**：早期 Vue Router 的 Hash 模式，就是通过监听 `window.onhashchange` 事件，在不刷新页面的情况下切换组件。 |

## 22、尾递归是什么？有哪些应用场景

要了解尾递归就得先搞明白递归是什么，递归是指：**在函数定义中使用函数自身的方法**，在函数的内部可以调用其他函数，如果在函数内部调用自己，那么这个函数就是递归函数。递归的核心思想是把一个大型的复杂问题层层转化为一个与原问题相似的规模较小的问题来求解。

一般来说，递归需要**边界条件**，这个边界条件决定了递归在什么时候结束，当边界条件不被满足的时候，递归前进，当边界条件被满足的时候，递归返回。这种传统的递归会在内存的**执行栈**（Call Stack)中不断压入新的栈帧以保存上下文，极易导致栈溢出也就是爆栈。

而尾递归，即在函数尾位置调用自身，尾递归是递归的一种特殊形式，它在普通递归的基础上多了这样两个特征

- 在尾部调用的是函数自身
- 可以通过优化，使得尾递归的计算只占用常量栈空间，所以尾递归不用担心爆栈的问题

在实际应用中，尾递归常用于处理深度未知的树形结构遍历、状态机流转，以及在纯函数式编程语言（如 Haskell）中替代传统的 `for/while` 循环。为了把普通递归改写成尾递归，我们通常会引入**累加器（Accumulator）模式**，将中间状态作为参数向下传递。"

简单来说，普通递归和尾递归的区别在于，普通递归在调用完自己后往往还要进行其他的操作，参考下面的代码，在调用完自己后还需要进行一个乘法，所以它在调用完过后还在内存中，如果这个递归的层级过大，就会导致内存爆炸。

```js
function factorial(n) {
  if (n === 1) return 1;
  // ⚠️ 注意：最后一步不是单纯的调用自身！
  // 它必须等待 factorial(n - 1) 计算出结果后，再乘以 n。
  return n * factorial(n - 1); 
}

factorial(5);
```

而尾递归，在调用完自身后，不会进行额外的操作，直接原地销毁，所以不用担心爆栈的问题

```js
// 引入 acc（累加器）来保存中间结果，初始值为 1
function tailFactorial(n, acc = 1) {
  if (n === 1) return acc;
  
  // 🌟 核心魔法：最后一步【仅仅】是调用自身。
  // 乘法运算已经在传参的这一刻执行完毕了！
  return tailFactorial(n - 1, n * acc); 
}
tailFactorial(5);
```



## 23、解构赋值是什么？为什么要用它？怎么用？

解构赋值是对赋值运算符的扩展。它是一种针对**数组**或者**对象**进行模式匹配，然后对其中的变量进行赋值。

在代码书写上简洁且易读，语义更加清晰明了；也方便了复杂对象中数据字段获取。

**代码极简**：消灭了满屏的 `user.xxx`，大大提升了代码的可读性。

**按需提取**：对象里有 100 个属性，我只需要 2 个，那我就只写那 2 个，干净利落。

**参数优雅**：在函数传参时，解构赋值是目前的绝对最佳实践。

| **场景**         | **语法格式**                | **实战用途**                                         |
| ---------------- | --------------------------- | ---------------------------------------------------- |
| **基础对象提取** | `const { a, b } = obj`      | 提取接口返回的 JSON 字段。                           |
| **对象重命名**   | `const { a: newA } = obj`   | 避免提取出来的变量名与当前作用域的变量名冲突。       |
| **对象默认值**   | `const { a = 10 } = obj`    | 兜底策略，防止后端没传字段导致页面报错 `undefined`。 |
| **基础数组提取** | `const [x, y] = arr`        | 解析图表坐标、经纬度 `[lng, lat]` 等有序数据。       |
| **数组跳跃提取** | `const [, , z] = arr`       | 只取数组特定位置的值，忽略前面的数据。               |
| **函数参数解构** | `function foo({ a, b }) {}` | 接收配置项对象时极其优雅，代码自文档化程度极高。     |

# CSS 面试题集锦

## 1、对盒子模型的理解

“面试官您好，关于对盒模型的理解，我是这样看的。在CSS布局中，浏览器会将每一个HTML元素都当作一个矩形的盒子来处理。这个盒子由内而外严格包含了四个部分：存放核心内容的**内容区（Content**）、内容与边框之间的**内边距（Padding）**、包裹元素的**边框（Border）**，以及控制元素与其他元素间距的**外边距（Margin）**。在实际开发中，我们最需要关注的是**标准盒模型**与**怪异盒模型**之间的差异，它们的核心区别在于***对元素尺寸的计算规则不同***。默认情况下，浏览器采用的是标准盒模型，此时我们给元素设置的宽度和高度仅仅作用于内容区。这意味着，如果后续给元素添加了内边距或边框，元素的实际总体积就会被撑大，这很容易导致原本排好的布局发生错位。为了解决这个问题，现代前端工程通常会通过CSS属性`box-sizing: border-box`，将元素切换为怪异盒模型。在这个模式下，我们设置的宽高就已经包含了内容区、内边距和边框的总和。增加内边距或边框只会向内挤压内容区，而不会改变元素在页面上占据的实际物理空间。这种'所见即所得'的特性，极大地降低了我们处理复杂响应式布局时的心智负担。”

### 盒模型分类

1. 标准盒模型：height和width不会把padding和margin一起计算

   盒子总高 = height + padding + border + margin

   盒子总宽 = width + padding + border + margin

2. IE怪异盒模型：height和width会把padding和margin一起计算

   盒子总高 = height + margin

   盒子总宽 = width + margin

   通过`box-sizing`属性，默认`content-box`为标准盒模型，`border-box`为怪异盒模型



## 2、css选择器有哪些？优先级？哪些属性可以继承？

关于CSS选择器，我们日常开发中主要依赖基础的标签选择器、类选择器、ID选择器，以及在处理嵌套DOM结构时常用的后代选择器、子选择器和相邻兄弟选择器。此外，还有用于捕捉用户交互状态的伪类选择器，比如`:hover`，以及用于插入或修饰特定文档片段的伪元素选择器，比如`::before`。当多个选择器作用于同一个元素时，浏览器会根据一套严格的权重算法来决定样式的优先级。在这套规则中，人为添加的`!important`标志拥有绝对的最高优先级，其次是直接写在HTML节点上的内联样式。在外部或内部样式表中，ID选择器的权重最高，紧接着是类选择器、属性选择器和伪类，最后才是基础的标签选择器和伪元素。如果多个选择器的权重计算结果完全一致，浏览器就会遵循'就近原则'，也就是代码位置靠后的样式会覆盖前面的。至于CSS属性的继承性，它的核心逻辑是减少代码冗余。通常来说，与文本排版和字体视觉强相关的属性都是默认可以被子节点继承的，比如`color`、`font-size`、`font-family`、`line-height`以及`text-align`等。而像盒模型相关的display、宽高、内外边距、边框，或者是背景色、定位这些控制区块物理形态和空间布局的属性，默认都是不可继承的，以防止父级布局意外污染子级结构。

| **优先级阶梯** | **选择器类型 / 规则**         | **权重数值**   | **常用语法示例**            | **核心特点与说明**                                           |
| -------------- | ----------------------------- | -------------- | --------------------------- | ------------------------------------------------------------ |
| **Top 1**      | **`!important` 规则**         | 无限大         | `color: red !important;`    | 它是超越一切规则的"核武器"，直接打破层叠规则。实际工程中应极力避免滥用，否则会导致样式难以维护和覆盖。 |
| **Top 2**      | **内联样式 (Inline)**         | `(1, 0, 0, 0)` | `<div style="color: red;">` | 直接挂载在 DOM 节点上，作用域极小，优先级仅次于 `!important`。 |
| **Top 3**      | **ID 选择器**                 | `(0, 1, 0, 0)` | `#header`                   | 针对页面中唯一的元素。由于权重过高，现代组件化开发（如 Vue/React）中通常不推荐用 ID 来写样式。 |
| **Top 4**      | **类选择器 (Class)**          | `(0, 0, 1, 0)` | `.btn-primary`              | 日常开发绝对的主力军，复用性极强。                           |
| **Top 4**      | **属性选择器**                | `(0, 0, 1, 0)` | `[type="checkbox"]`         | 常用于表单元素的精准控制。                                   |
| **Top 4**      | **伪类选择器 (Pseudo-class)** | `(0, 0, 1, 0)` | `:hover`, `:nth-child(2)`   | 用于描述元素的特定交互状态或DOM树结构位置。                  |
| **Top 5**      | **标签选择器 (Type)**         | `(0, 0, 0, 1)` | `div`, `span`, `p`          | 权重极低，通常只在做 CSS Reset（全局样式重置）或基础排版时使用。 |
| **Top 5**      | **伪元素 (Pseudo-element)**   | `(0, 0, 0, 1)` | `::before`, `::after`       | 用于在 DOM 节点前后凭空插入虚拟元素，常用于清除浮动或添加装饰性图标。 |
| **Top 6**      | **通配符选择器**              | `(0, 0, 0, 0)` | `*`                         | 匹配页面所有元素，性能开销较大，且权重为0。                  |
| **Top 6**      | **关系连接符**                | `(0, 0, 0, 0)` | `>`, `+`, `~`, `空格`       | 它们只负责表达节点间的父子或兄弟关系，**本身不增加任何权重**。 |
| **最低**       | **继承的样式**                | 无权重         | 父级设置的 `color`          | 继承来的样式优先级极其卑微，甚至连通配符 `*` 都能轻松覆盖它。 |

## 3、说说em/px/rem/vh/vw区别?

“面试官您好。首先这些在CSS中都是长度单位，它们的核心区别在于它们是绝对单位还是相对单位，以及它们相对的'参照物'究竟是谁。首先，`px`是我们最常用的绝对物理单位，它代表屏幕上的一个像素点，无论页面或父元素怎么变化，它的大小始终固定，通常只用于边框线条或是需要极其精确控制的微小间距。相比之下，`em`是一个相对单位，它相对于当前元素或父元素的字体大小来计算。虽然灵活，但`em`存在一个致命的缺陷：如果在多层嵌套的DOM结构中都使用了`em`，它的计算结果会呈指数级失控，导致样式极难维护。为了解决这个痛点，CSS3引入了`rem`，也就是Root em。它永远且唯一地参照HTML根节点的字体大小进行等比例计算。这使得`rem`彻底摆脱了DOM树嵌套层级的干扰，成为了目前移动端自适应适配方案中的绝对主力。最后是`vh`和`vw`，它们被称为视口单位，分别代表浏览器当前可视区域高度和宽度的百分之一。与传统的百分比单位严格受限于父容器尺寸不同，视口单位可以完全无视DOM结构的嵌套深度，直接霸道地获取屏幕的真实物理尺寸，这在开发全屏铺满的引导页、居中弹窗或是大屏数据看板时，是最直接高效的手段。”

| **长度单位** | **单位类型**                                           | **参照物 (计算基准)**               | **核心特点与表现**                                           | **典型应用场景 (结合表单设计器)**                            |
| ------------ | ------------------------------------------------------ | ----------------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **`px`**     | 绝对单位（为了区分这里说是绝对单位，但是其实不严谨！） | 设备的物理像素点                    | 尺寸固定死板，无论屏幕或父容器怎么变，它都不变。             | 边框线粗细（如 `1px solid`）、阴影偏移量、表单组件绝对定位拖拽时的精确坐标。 |
| **`em`**     | 相对单位                                               | **自身或直系父元素**的 `font-size`  | 存在致命的"级联放大"效应，DOM嵌套越深，尺寸计算越容易彻底失控。 | 局部文本排版，例如段落首行缩进（`text-indent: 2em` 永远缩进两个字符的宽度）。 |
| **`rem`**    | 相对单位                                               | **根节点 (`<html>`)**的 `font-size` | 全局唯一参照物。改动根节点字体大小，全页面的 `rem` 尺寸会随之等比联动缩放。 | 移动端整体响应式布局的主力方案（常配合 JS 动态计算根节点字体大小）。 |
| **`vw`**     | 视口单位                                               | **浏览器可视窗口宽度**的 1%         | 绝对霸道，直接穿透所有 DOM 父容器，只看当前浏览器屏幕有多宽。 | 全屏宽度的横幅区块、大屏数据看板的基础宽度单位。             |
| **`vh`**     | 视口单位                                               | **浏览器可视窗口高度**的 1%         | 同上，只看当前浏览器屏幕可视区域有多高。                     | 首屏引导页（`height: 100vh` 铺满屏幕）、表单设计器中保证画布区域占满剩余高度。 |

⚠️注意：我们往往习惯性地把 `px` 称作绝对单位，因为它不受 DOM 树嵌套层级或父元素字体大小的影响。但在浏览器的底层渲染引擎中，`px` 相对的参照物是设备的显示分辨率以及用户视网膜与屏幕之间的物理观看距离，在这种层面px其实是相对单位。最严谨的结论是：`px` 在前端组件布局的语境下充当了绝对单位，但其底层物理本质绝对是一个相对单位。



## 4、说说设备像素、css像素、设备独立像素、dpr、ppi 之间的区别？

"面试官您好。这五个概念构成了现代屏幕显示的底层坐标系，它们之间的关系其实是一套从物理硬件到软件逻辑的映射体系。首先最底层的基石是'设备像素'，也就是物理像素，它是屏幕面板上真实存在的、不可再分的最小发光二极管，在屏幕出厂那一刻就绝对固定了。但是随着技术发展，屏幕的物理发光点越来越密集，如果系统直接用物理像素来渲染UI，那么在高清屏上所有的按钮和文字都会变得极度细小、难以点击。为了解决这个问题，操作系统抽象出了一层'设备独立像素'，也就是逻辑像素。它是一个虚拟的坐标单位，系统通过底层的换算，保证同一个逻辑像素在不同分辨率的手机上，呈现出的真实物理尺寸在视觉上基本一致。而我们在前端代码中写的'CSS像素'，本质上就是Web视域下的设备独立像素。在浏览器默认百分之百缩放、且没有改变视口设置的情况下，一个CSS像素严格等于一个设备独立像素；但当用户双指放大页面时，一个CSS像素所覆盖的设备独立像素面积就会随之成倍增加。连接这物理和逻辑两套体系的桥梁正是'DPR'，即设备像素比。它定义了在一个单一方向上，一个逻辑像素究竟由几个物理像素来负责渲染。比如在苹果的Retina高清屏上，DPR通常是2或者3，这意味着在代码里写的一个CSS像素，实际上是由4个或9个物理发光点共同点亮的，所以画面看起来极其细腻。最后，决定屏幕到底采用多大DPR的物理根源在于'PPI'，也就是每英寸像素数。它反映了屏幕面板真实的物理像素密度，PPI数值越高，代表物理发光点排列越紧密，手机厂商为了保证UI不缩水，就会在系统层面给这块屏幕分配更高的DPR值。"

| **概念**         | **英文简称**         | **属性归属**    | **核心作用与表现**                                           |
| ---------------- | -------------------- | --------------- | ------------------------------------------------------------ |
| **设备像素**     | DP (Physical Pixels) | 纯硬件          | 屏幕上真实的微小发光点（如 1080x1920 说的就是这个）。        |
| **PPI**          | PPI                  | 纯硬件          | 物理像素密度。数字越大，屏幕越清晰，没有颗粒感。             |
| **设备独立像素** | DIP / PT / DP        | 操作系统层面    | 虚拟的逻辑坐标系，保证不同设备的 UI 控件物理大小基本一致。   |
| **CSS 像素**     | PX                   | 浏览器/Web 层面 | 前端代码中使用的逻辑单位。受页面缩放比例控制。               |
| **DPR**          | DPR                  | 软硬结合的桥梁  | `window.devicePixelRatio`。决定了一个逻辑像素用几个物理像素来画。 |



## 5、css中，有哪些方式可以隐藏页面元素？区别?

"面试官您好。在CSS中隐藏页面元素，我们最常用的核心方案有三种，它们的主要区别在于是否保留物理占位、是否响应鼠标交互以及对浏览器渲染性能的影响。最彻底的隐藏方式是 `display: none`，一旦设置，元素及其所有子节点都会从浏览器的渲染树中完全剥离，它不再占据任何页面空间，也绝对无法触发任何交互事件，但这会导致页面布局的重新计算，也就是引发性能开销较大的'重排'（Reflow）。如果我们希望元素在视觉上消失，但依然保持它原本在文档流中的骨架和物理占位，通常会选择 `visibility: hidden`，此时元素虽然隐身且占据着空间，但它同样屏蔽了所有的鼠标事件，这种方式改变的只是外观，只会引发'重绘'（Repaint）。如果我们的场景需要配合极其丝滑的淡入淡出动画，那么 `opacity: 0` 是不二之选。它仅仅是将元素的透明度降到了零，元素依然原封不动地停留在原地占据空间。但它最特殊的一点是，处于透明状态的元素默认依然可以完美接收并触发原本绑定的所有点击或悬浮事件。在实际工程中，比如开发画布区域的动态组件时，如果不想破坏整体的网格流式布局，我们可能会避免使用 `display: none`，而是通过透明度配合绝对定位，或者直接利用 `pointer-events: none` 来彻底剥夺隐身元素的交互能力。此外，针对视障用户的无障碍访问需求，我们还会使用 `position: absolute` 将元素移到 `-9999px` 的屏幕之外，这样普通人看不见，但屏幕阅读器依然能精准抓取到文本内容。"

| **隐藏方案核心代码**                 | **隐藏的底层原理 (它到底干了什么？)**                        | **干预的渲染阶段**   | **是否占据物理空间** | **是否响应交互** | **核心实战应用场景**                                         |
| ------------------------------------ | :----------------------------------------------------------- | -------------------- | -------------------- | ---------------- | ------------------------------------------------------------ |
| **`display: none`**                  | **渲染树拦截**：浏览器在将 DOM 树与 CSSOM 树合并成 Render Tree（渲染树）时，直接忽略该节点及其所有子节点。它根本没有被交到排版引擎手里。 | 构建渲染树阶段       | **否**               | **否**           | 彻底不显示的元素，如切换 Tab 标签页、通过条件判断移除的组件  |
| **`visibility: hidden`**             | **绘制阶段跳过**：元素正常进入渲染树，也正常参与盒模型的布局计算，但在最终往屏幕上"涂颜色"的绘制阶段，浏览器跳过了属于该元素的像素点绘制。 | 绘制阶段 (Paint)     | **是**               | **否**           | 需要保留元素位置以维持布局稳定，且后续可能只让其子元素显示的场景。 |
| **`opacity: 0`**                     | **Alpha 通道归零**：元素不仅参与布局，也正常进行了颜色绘制，但在最终的图层合成时，GPU 将该图层的透明通道（Alpha）设置为了 0，变成了"绝对透明的玻璃"。 | 图层合成 (Composite) | **是**               | **是**           | 需要配合 `transition` 实现淡入淡出视觉动画的弹窗或提示框。   |
| `position: absolute` `left: -9999px` | **视口坐标系放逐**：元素被正常渲染，拥有完整的尺寸和内容。但系统根据绝对定位坐标，将其强行渲染在了当前浏览器屏幕可视窗口范围之外的遥远位置。 | 布局阶段 (Layout)    | **否** (脱离正常流)  | **否** (点不到)  | **无障碍访问(a11y)与SEO**：视觉上不可见，但屏幕阅读器和爬虫依然能读取文本内容。 |
| `height: 0` `overflow: hidden`       | **盒模型尺寸坍缩+裁剪**：将盒模型的高度参数强制设为0。由于内容可能溢出这个0高度的盒子，`overflow: hidden` 指示渲染引擎在绘制时，裁剪掉所有超出0高度边界的像素。 | 布局阶段 + 绘制阶段  | **否**               | **否**           | 手风琴折叠面板、下拉菜单的平滑展开与收起动画。               |
| **`transform: scale(0)`**            | **渲染矩阵坍缩**：元素本身的物理尺寸依然存在，但在 GPU 进行图层合成时，应用了一个缩放矩阵，将该图层的所有视觉像素点向中心挤压成了一个面积为 0 的数学点。 | 图层合成 (Composite) | **是**               | **否**           | UI 元素以自身中心点缩放消失的动效（如点赞红心收缩消失）。    |
| **`clip-path: circle(0)`**           | **绘制蒙版裁剪**：给元素覆盖了一层面积为 0 的极小矢量蒙版。浏览器只绘制蒙版内部的像素，蒙版外全透明，导致元素彻底不可见。 | 绘制阶段 (Paint)     | **是**               | **否**           | 带有矢量遮罩裁剪效果的高级过渡动画。                         |



## 6、讲讲重绘和重排

"面试官您好。在浏览器的渲染机制中，**重排（Reflow）**和**重绘（Repaint）**是决定页面性能的核心因素。<u>重排是指当 DOM 元素的几何属性，比如宽高、内外边距、隐藏状态或是定位发生变化时，浏览器必须重新计算元素的几何空间位置，并重建渲染树的过程</u>。<u>重绘则是指元素的非几何视觉属性，比如背景色、文字颜色或者阴影发生改变，浏览器只需要把新的样式重新绘制到屏幕上的过程。</u>从底层开销上看，重排的代价极其昂贵，它甚至会引发整个页面的连锁重新布局，而且发生**重排必定会触发重绘，但重绘不一定会引发重排**。在实际业务开发中，最经典的性能灾难叫做'强制同步布局'或'布局抖动'，也就是在 JavaScript 中通过循环频繁交替地读取和修改 DOM 的布局属性，导致浏览器自身的渲染队列优化机制失效，被迫反复执行重排逻辑。为了规避这些问题，我们通常会采用批量修改 CSS 类名而不是逐条修改内联样式、使用 DocumentFragment 批量构建和插入 DOM 节点，或者在做复杂动画时优先利用 CSS3 的 transform 和 opacity 属性，将元素提升为独立的合成层，从而直接利用 GPU 硬件加速，彻底避开主线程的重排与重绘计算。"

### 场景：批量修改样式的正确姿势

当你需要同时修改一个元素的多个尺寸和样式时。

**❌ 糟糕的代码（触发多次重排/重绘）：**

```js
const box = document.getElementById('my-box');
// 浏览器本来有队列优化机制，但老旧浏览器或复杂上下文中
// 这种逐行修改内联样式的行为，极易触发多次重排
box.style.width = '200px'; 
box.style.height = '200px';
box.style.padding = '10px';
box.style.border = '1px solid red';
```

**✅ 优秀的代码（合并为 1 次重排）：**

```js
const box = document.getElementById('my-box');
// 方式 1：通过 cssText 一次性修改
box.style.cssText = 'width: 200px; height: 200px; padding: 10px; border: 1px solid red;';

// 方式 2：现代框架（Vue/React）和原生开发中最推荐的做法：切换 Class
// CSS: .active-box { width: 200px; height: 200px; padding: 10px; border: 1px solid red; }
box.classList.add('active-box'); 
```



## 7、谈谈你对BFC的理解？

"面试官您好。BFC 的全称是块级格式化上下文（Block Formatting Context），它是 CSS 页面布局中的一个极其核心的底层机制。通俗地讲，BFC 就是页面上的一个被彻底隔离的独立渲染区域。在这个区域内部，子元素会按照标准的块级规则进行垂直排列，而且最关键的特性是，BFC 内部的元素无论怎么布局变换，都绝对不会越界去影响到区域外部的元素，反之亦然。在日常开发中，我们无法直接声明一个叫做 BFC 的属性，而是需要通过给元素设置特定的 CSS 样式来隐式地触发它，比如将父元素的 `overflow` 属性设置为 `hidden` 或 `auto`，又或是将其定位 `position` 设置为 `absolute` 或 `fixed`，或者使其浮动 `float`，以及将 `display` 设置为 `flex`、`grid` 或 `inline-block` 等。掌握 BFC 的核心价值，在于它能完美解决我们构建复杂页面时的三大经典顽疾：第一，利用 BFC 的隔离特性，可以打破相邻块级元素在垂直方向上的外边距塌陷重叠现象；第二，触发 BFC 的父容器可以计算并包裹住内部浮动的子元素，从而彻底解决父级高度坍缩的问题；第三，利用 BFC 区域不会与浮动元素发生重叠的特性，我们可以非常优雅地实现左侧宽度固定、右侧内容自适应的经典两栏布局。"

## 8、说说flexbox（弹性盒布局模型）,以及适用场景？

"面试官您好。Flexbox，也就是弹性盒布局模型，是 W3C 提出的一种专门为现代 UI 设计的布局方案。它的核心设计哲学是提供一种高效的方式来动态分配容器内部的空间，并精准控制子元素的对齐方式，即使这些子元素的初始物理尺寸是未知的或是动态变化的。与传统的基于盒模型和浮动机制不同，Flexbox 本质上是一种一维布局模型，它的排版引擎严格围绕着'主轴'和'交叉轴'这两条相互垂直的坐标线来展开。只要给父容器开启了 `display: flex`，我们就可以彻底告别过去为了居中或对齐而写的各种奇技淫巧，利用配套属性极度优雅地控制子元素的排列方向、换行规则以及空间瓜分比例。在实际的业务场景中，Flexbox 的适用范围极广。比如在我目前负责的表单设计器项目中，不管是顶部工具栏里各种操作按钮的垂直居中与两端对齐，还是左侧物料面板里组件列表的流式换行排列，甚至是画布区域内表单标签（Label）与输入控件（Input）的动态自适应并排，全都是重度依赖 Flexbox 来完成的。概括来说，只要涉及到单行或单列的线性组件排版，或者是需要元素能够根据屏幕或外层容器的可用空间进行自动弹性伸缩的局部布局，Flexbox 都是目前工程实践中的绝对首选。"

## 9、介绍一下grid网格布局

"面试官您好。CSS Grid 网格布局是目前 Web 前端领域唯一原生支持的二维布局系统。与 Flexbox 只能处理单行或单列的一维流式布局不同，Grid 能够同时在水平和垂直两个维度上对页面空间进行极其精确的切割和划分。开启 `display: grid` 后，我们主要通过定义行和列的轨道，也就是 `grid-template-rows` 和 `grid-template-columns`，在父容器内部构建出一个无形的坐标系。它引入了极其强大的 `fr` 剩余空间分配单位，以及 `minmax()` 等响应式函数，让我们无需精细计算百分比就能实现复杂的自适应排版。在实际工程中，我会严格界定这两大布局系统的边界：当需要处理局部组件的动态对齐、或者内容数量不确定的单向排列时，我首选 Flexbox；但如果是构建页面的宏观骨架，或者是处理类似复杂表单中需要精确跨行、跨列的网格化内容时，我会毫不犹豫地使用 Grid 布局。特别是配合 `grid-template-areas` 属性，我们可以直接用语义化的字符串在 CSS 中'画'出整个页面的结构，这让代码的后期维护性和可读性得到了质的飞跃。"



## 10、元素水平垂直居中的方法有哪些？如果元素不定宽高呢？

"面试官您好，在现代前端开发中，实现元素水平垂直居中，尤其是针对宽高完全不确定的动态内容，有几套标准的方案。首选方案是<u>利用Flex布局</u>，只需要在父容器上声明`display:flex`，并配合`justify-content: center`以及`align-items:center`，浏览器就会自动接管剩余空间的分配，完美实现子元素在交叉轴和主轴上的绝对居中。同时，如果项目的兼容性允许，也可以<u>采用CSS Grid</u>，只需给父级设置 `display: grid` 以及 `place-items: center`即可。当然，如果需要处理一些脱离文档流的弹窗或悬浮层，我会采用绝对定位配合 Transform 的方案。具体做法是父元素相对定位，子元素绝对定位后将其 `top` 和 `left` 属性都推至 50%，此时子元素的左上角会处于中心点，接着再利用 `transform: translate(-50%, -50%)` 将子元素基于自身实际渲染的宽高向回拉扯一半的距离。"

## 11、CSS3是什么，增加了哪些特性

### 2、你在做项目过程中，遇到最具挑战性的问题

"在这个我自己做的低代码开源练习项目中，我所遇到的最棘手的问题就是：多层级嵌套容器所产生的一系列问题。

多层级嵌套就是：卡片里面套栅格，栅格里面套标签页，标签页里面再套图表和输入框，这种就是多层级嵌套。

这就会导致：**选中穿透**和**组件量堆积后的卡顿**

第一个是**选中穿透**。因为 DOM 原生的事件冒泡机制，当用户想选中最内层的输入框时，点击动作会疯狂向上冒泡，导致右侧属性面板总是错误地跳到最外层'卡片'的配置。 为了实现'指哪打哪'的精准选中，我没有去写复杂的全局状态判断，而是直接在引擎渲染的每个物料外围，精准绑定了 `@click.stop` 修饰符，阻止事件冒泡。只要鼠标悬停在最内层，立刻切断事件向上冒泡，这就保证了组件能精准落入指定的层级。

其次是解决卡顿问题。排查后我发现，问题出在我对 Vue3 响应式 API 的滥用上。Vue 默认的 `reactive` 会对传入的复杂对象进行深度的递归遍历，给每一个节点都挂载 Proxy 代理。拖拽时频繁触发这个过程，开销太大了。 所以我对数据状态做了一次拆分：对于左侧菜单里的原始物料模板，以及存放在内存里的撤销操作快照，这类不需要局部修改的庞大静态数据，我全部改用了 `shallowRef`。它只监听第一层的引用变化，直接省去了海量深层代理的性能开销。然后在真正把组件拖入画布的那一刻，我再用 `Lodash` 进行深拷贝生成新 ID，切断引用污染。"

**实在的结果与收获 (Result)** "经过这两步调整，即使我故意在画布里套很多层，拖拽起来也基本感受不到掉帧了，也不会再出现错位的问题。 虽然这只是个人的练手项目，但这个排坑的过程，让我对 Vue3 底层的 `Proxy` 机制、`ref` 与 `shallowRef` 的实际性能差异，以及事件流的控制，都有了非常直观的实战体会，而不再是仅仅停留在看文档的层面。"



### 3、事件循环以及代码输出考察

"因为 JavaScript 是单线程的，为了防止耗时任务阻塞主线程，JS 引擎采用了一套基于事件循环的异步并发机制。

在执行代码时，任务会被分为同步任务和异步任务。同步任务（像普通的 `console.log`、变量赋值、数学计算）直接在主线程的调用栈中执行。而异步任务有了结果后，会被分发到两个不同的队列中：**微任务队列**（像 `Promise.then`）和**宏任务队列**（像 `setTimeout`）。

注意：遇到 `async/await` 不要慌！在 JS 引擎眼里，`await` 后面的那一**整行代码**是同步执行的（相当于 `new Promise` 里的代码）；而 `await` **下面**的所有代码，会被 JS 引擎偷偷打包，塞进一个 `.then()` 里面，也就是变成了**微任务**。

执行的优先级是：**同步任务 --> 异步任务中的微任务 ---> 异步任务中的宏任务**

事件循环的运行机制是：首先执行并清空当前调用栈里的所有同步代码；同步代码执行完毕后，JS 引擎会去检查微任务队列，并且**清空整个微任务队列**里的所有任务；微任务清空后，引擎会去宏任务队列里取出**第一个宏任务**推入主线程执行。执行完这一个宏任务后，引擎会**再次回头去清空此时产生的微任务**。

这样周而复始的不断交替检查和执行的过程，就是事件循环。" 	

这个知识点，通常通过代码段的输出的方式来考察，比如：

```js
console.log('1');

// setTimeout 中的代码是异步任务中的宏任务
setTimeout(() => {
  console.log('2');
  Promise.resolve().then(() => {
    console.log('3');
  });
}, 0);

// 注意：promise内部的代码是同步任务，.then里面的代码是异步任务中的微任务
new Promise((resolve) => {
  console.log('4');
  resolve();
}).then(() => {
  console.log('5');
  setTimeout(() => {
    console.log('6');
  }, 0);
});

console.log('7');

// 1、4、7、5、2、3、6
```



### 4、Vue的双向绑定以及Vue2、Vue3分别是怎么实现双向绑定的

"面试官您好。说到双向绑定， 它的核心思想就是 MVVM 架构：视图（View）和数据模型（Model）是实时同步的。用户在输入框里打字，JS 里的变量自动更新；反过来，我们用代码修改了变量，页面上的文字也瞬间改变。

"但实际上，双向绑定并不是什么魔法。**我们在代码里高频使用的 `v-model`，本质上就是一个'语法糖'。 如果在 Vue 的模板编译阶段把它拆开来看，它其实就是'单向属性绑定（v-bind）'和'事件监听（v-on）'的结合体。**具体来说，它是由两部分闭环组成的：

**第一部分是'视图到数据（View -> Model）'**：这对应着语法糖里的**事件监听**部分。原理很简单，就是底层帮我们自动绑定了 DOM 事件（比如原生输入框的 `@input` 事件，或者 Vue3 组件的 `@update:modelValue` 事件）。一旦监听到用户在页面上的输入，就立刻拿到最新值，去修改 JS 里对应的数据。

**第二部分，也是整个 Vue 框架最核心的魔法，也就是'数据到视图（Model -> View）'**：这对应着语法糖里的**属性绑定**部分（比如把变量绑定给 `:value` 或 `:modelValue`）。但这里的关键问题是，当我们用 JS 代码在后台改变了一个变量的值，Vue 是怎么立刻知道数据变了，并且精确地去更新绑定了这个变量的那一块 DOM 呢？

这就引出了 Vue 最核心的底层机制：**数据劫持（拦截器）结合发布订阅模式（观察者）**。 Vue 在初始化时，会给数据对象安插拦截器。当组件渲染读取数据时，拦截器会把这个组件记录下来（收集依赖）；当数据被修改时，拦截器立刻察觉，并通知所有依赖这个数据的组件去重新渲染（派发更新）。

**不过，说到这个'底层拦截器'，Vue 2 和 Vue 3 的实现有着本质的代差，尤其是在处理数组的时候。**

在 **Vue 2** 中，底层使用的是 `Object.defineProperty`。它的致命缺陷是只能拦截对象里已经存在的具体属性。所以，如果我们直接通过下标修改数组（比如 `arr[0] = 1`）或者修改数组长度，它是完全拦截不到的。为了填补这个漏洞，Vue 2 的源码被迫采用了一种 Hack 手法，**重写了数组原型的 7 个变异方法（比如 push、splice 等）**，通过拦截这几个方法的调用来手动触发视图更新。

而在我目前主攻的 **Vue 3** 项目中，底层全面拥抱了 ES6 的 **`Proxy`**。 Proxy 是降维打击，它不再是死盯着某个具体属性，而是**直接给整个对象或整个数组罩上了一层代理结界**。这意味着，无论是修改数组下标、改变数组长度，还是给对象动态新增未声明的属性，Proxy 的 `set` 拦截器都能在底层完美捕捉到。这不仅彻底丢掉了 Vue 2 重写数组方法的历史包袱，在处理像低代码平台那种深层嵌套的复杂 JSON 树时，性能和代码的纯粹性也得到了极大的提升。"



### 5、前端缓存用过哪些

"面试官您好，关于前端缓存，从四个维度来统筹考虑：

首先，在**网络层**，我会配置合理的 HTTP 缓存策略。比如对打包后的 JS/CSS 静态资源开启 `Cache-Control` 强缓存；对于 `index.html` 这种入口文件，利用 `ETag` 配合后端做协商缓存，极大降低首屏加载的网络带宽。

其次，在**浏览器存储层**，除了常规的使用 `localStorage` 存储用户 Token 和身份标识外，面对需要落地的复杂结构化大文件（比如低代码平台中庞大复杂的页面 JSON 配置草稿），我会优先选用 `IndexedDB` 来突破 5MB 的存储限制，防止内存溢出。

第三，在**框架内存层**，我会大量使用 Vue 的 `<keep-alive>` 来缓存多级路由切换时的列表状态；同时配合 `Pinia` 作为全局状态中心，对高频读取的字典表和权限角色树做内存级缓存。

最后，在**接口请求层**，为了追求极致体验，我会在 Axios 的底层封装中，利用 `Map` 结构缓存正在 Pending 的 Promise 请求，用来防御用户手抖造成的接口高频并发。

从网络拦截、持久化存储、框架内存再到请求防抖，这就是我在实际业务中对前端缓存体系的整体运用。"

### 2、git的版本管理

"在准备推送到远端之前，我绝不会盲目 push。我会先执行 **`git fetch origin main`** 获取最新进度，然后执行 **`git rebase origin/main`** 进行本地变基。 如果遇到冲突，我会利用 VS Code 解决后，执行 **`git add .`** 和 **`git rebase --continue`**。这样处理完之后再 **`git push`** 发起 PR，就能保证主干的 Git 历史是一条毫无分叉的直线。"

#### 2.1、第一阶段：本地开发与拦截（切分支 -> 提交）

保证基于最新代码开发，并触发本地质检

```cmd
# 1. 永远先切回主干，拉取全团队最新的可用代码
git checkout main
git pull origin main

# 2. 绝对不在 main 上干活，切出一个你的专属特性分支
git checkout -b feat/lowcode-panel

# --- 此处疯狂写代码 ---

# 3. 代码写完，准备提交
git add .

# 4. 触发 Husky 和约定式提交（不要写 git commit -m "update"）
git commit -m "feat: 新增低代码左侧拖拽物料库"
# 此时终端会自动跑 ESLint，如果有错，这个命令会直接报错并终止！
```

#### 2.2、第二阶段：推送前的防碰撞（拉取 -> 变基）

保证你的代码和别人刚刚提交的代码不打架。

```cmd
# 1. 把远端主干的最新变化"偷偷"拉取下来（只下载，不合并）
git fetch origin main

# 2. 极其核心的动作：执行变基！把你的提交挪到别人最新提交的后面
git rebase origin/main

# --- 假设天塌了，有冲突了 ---
# 终端会提示你哪些文件 conflict 了。你去 VS Code 里点 "Accept Current" 等把冲突解掉。

# 3. 冲突解完后，告诉 Git 你处理好了
git add .
git rebase --continue 
# （注意：变基解冲突千万不要敲 git commit！）
```

#### 2.3、推送到云端并发起 PR

```cmd
# 1. 把你在本地收拾得极其干净的分支，推送到远端
# （-u 的意思是把本地分支和远端分支绑定，以后直接 git push 就行）
git push -u origin feat/lowcode-panel
```



### 3、数组的处理 —— 如何转换与过滤后端数据

**传统方法**

```js
// 1. 模拟后端返回的原始数据
const rawBackendData = [
  { id: "1001", name: "高级机械键盘", price: 29900, status: 1, createTime: 1704067200000, isDeleted: 0, user: { nickname: "张三", level: "VIP" } },
  { id: "1002", name: "无线鼠标", price: 9900, status: 2, createTime: 1704153600000, isDeleted: 0, user: { nickname: "李四", level: "Normal" } },
  { id: "1003", name: "已取消订单", price: 5000, status: 3, createTime: 1704240000000, isDeleted: 1, user: { nickname: "王五", level: "Normal" } },
  { id: "1004", name: "超大鼠标垫", price: 4900, status: 1, createTime: 1704326400000, isDeleted: 0, user: { nickname: "赵六", level: "VIP" } }
];

// 2. 配置化：状态字典（策略模式，方便以后扩展状态）
const STATUS_CONF = {
  1: { label: '待发货', color: 'blue' },
  2: { label: '已完成', color: 'green' },
  3: { label: '已取消', color: 'gray' }
};

/**
 * 核心处理函数（传统方法）
 */
const getCleanData = (data) => {
  // 1. 必须手动创建一个"空篮子"来装结果
  const result = [];

  // 2. 判空防错
  if (!Array.isArray(data)) return result;

  // 3. 开始手动循环
  for (const item of data) {
    // 【第一步：手动判断过滤】
    if (item.isDeleted === 0) {
      
      // 【第二步：手动进行转换】
      const statusInfo = STATUS_CONF[item.status] || { label: '未知', color: 'black' };
      
      // 【第三步：手动组装新对象】
      const newItem = {
        id: item.id,
        goodsName: item.name,
        priceText: `¥${(item.price / 100).toFixed(2)}`,
        statusLabel: statusInfo.label,
        statusColor: statusInfo.color,
        date: new Date(item.createTime).toLocaleDateString(),
        userName: item.user?.nickname || '匿名用户'
      };

      // 【第四步：手动塞进篮子】
      result.push(newItem);
    }
  }

  return result;
};

// 执行转换
const result = getCleanData(rawBackendData);

// 打印结果看看
console.table(result);
```

**链式调用**

```js

 * 核心处理函数（链式调用）
 */

    // 【第一步：清洗/过滤】只保留没被删除的数据

      // c. 返回全新的对象（不修改原数据，保持不可变性）
      // 如何处理数据关键就是这里
      return {
        id,
        goodsName,
        // 金额：分转元，保留两位小数
        priceText: `¥${(price / 100).toFixed(2)}`,
        // 状态：映射文字和颜色
        statusLabel: statusInfo.label,
        statusColor: statusInfo.color,
        // 时间：利用 Date 对象格式化
        date: new Date(createTime).toLocaleDateString(),
        // 扁平化：利用可选链 ?. 提取嵌套属性，防止 user 缺失报错

      };
    });
};
```

当实际项目中拿到后端数据时，通常需要经过处理后才能进行使用，我的思路是，先用**filter**方法过滤清洗一遍数据，最后使用**map**通过**解构赋值**加工成符合要求的数据



### 4、Get和Post的区别

get和post都是Http协议中发送请求的方法。本质上都是`TCP`链接，并无差别

`GET`方法请求一个指定资源的表示形式，使用GET的请求应该只被用于获取数据

`POST`方法用于将实体提交到指定的资源，可能导致在服务器上的状态变化或**副作用**

#### 区别

| **维度**         | **GET 请求**                                       | **POST 请求**                                |
| ---------------- | -------------------------------------------------- | -------------------------------------------- |
| **回退/刷新**    | **无害**。不会重新发送请求。                       | **有害**。浏览器会提示将再次提交表单。       |
| **收藏/书签**    | **支持**。URL 包含完整参数，可被收藏。             | **不支持**。参数不在 URL 中，不可被收藏。    |
| **缓存 (Cache)** | **主动缓存**。除非手动设置不缓存。                 | **不缓存**。除非手动设置缓存。               |
| **编码方式**     | **URL 编码**。 (application/x-www-form-urlencoded) | **支持多种**。 (JSON, FormData, XML, 文本等) |
| **历史记录**     | 参数完整保留在浏览器历史记录中。                   | 参数不会被保留。                             |
| **参数长度**     | **有限制**。受浏览器 URL 长度限制（约 2KB）。      | **无限制**。理论上没有大小限制。             |
| **数据类型**     | 只接受 **ASCII 字符**。                            | **没有限制**。支持二进制、大文件、JSON 等。  |
| **安全性**       | **较低**。参数暴露在 URL 上，不能传敏感信息。      | **相对较高**。参数在请求体中，不会直接暴露。 |
| **传输位置**     | 参数拼接在 **URL** 后面。                          | 参数存放在 **Request Body** (请求体) 中。    |



### 5、常见的水平居中方案

我会首先推荐 Flexbox 和 Grid 这种现代布局方案，比如在父元素上直接使用 `display: flex; align-items: center; justify-content: center;` 或者是更极致的 `display: grid; place-items: center;`，这两者是目前 Vue 3 项目中最标准、代码量最少、且最符合响应式设计的做法。

如果涉及到像低代码画布中的组件定位，或者全局弹窗这种需要脱离文档流的场景，我会转向绝对定位方案。对于宽高动态变化的组件，我会用 `top: 50%; left: 50%` 配合 `transform: translate(-50%, -50%)`。之所以强调 `transform`，是因为它是相对于组件自身宽高进行偏移的，而且它在浏览器渲染层面走的是 GPU 复合层，不会引起重排，性能非常出色。

当然，如果是那种宽高完全固定的元素，我会利用把上下左右全部设为 0 然后 `margin: auto` 的经典技巧，这在处理固定比例的登录框时极其稳健。最后对于最简单的单行文本，我会顺带提一下 `line-height` 等于容器高度的轻量级方案。

| **方案**                   | **完整代码实现 (父 + 子)**                                   | **适用场景**                                     | **优缺点分析**                                               |
| -------------------------- | ------------------------------------------------------------ | ------------------------------------------------ | ------------------------------------------------------------ |
| **Flexbox**                | `.parent { display: flex; align-items: center; justify-content: center; }` | **现代网页布局首选**。同时实现水平垂直居中。     | **优点**：最通用，代码极简，支持响应式。 **缺点**：极老旧浏览器（IE9-）不兼容。 |
| **Grid**                   | `.parent { display: grid; place-items: center; }`            | **追求极致简洁**。只需要在父元素写两行。         | **优点**：代码量最少。 **缺点**：语义上更适合网格布局，大面积使用需考虑性能。 |

### 6、多个页面请求同样的数据，如何减少性能消耗？

在实际的业务中，比如用户信息或者全局配置字典，很多页面都会用到。如果每个页面进去都重新发送一次请求，不仅浪费用户流量，更会给后端服务器造成巨大压力。通常可以采用**多级缓存策略**来解决这个问题。

最常用的可以称之为第一级缓存的是：**全局状态管理缓存**。例如Vue项目，可以用Pinia维护一个全局的store。在Action触发请求逻辑之前，先进行一次判断：即**检查Store中是否已经存在该数据**，只要缓存中存在该数据，就直接返回该数据。

如果遇到更加极端的情况，比如一个页面同时有多个组件在同一时刻被渲染，它们都需要触发同一个初始化请求。这个时候前面的那种判断策略就失效了，因为这个时候缓存还没来得及更新。针对这种并发场景，我会从 **Axios 拦截器**这一层进行优化。我会维护一个 `pendingPromise` 的对象，以请求的 URL 和参数作为唯一标识。当第一个请求发出后，我会把它的 Promise 存起来。后续四个请求进来时，发现已经有相同的请求在排队，就直接返回那第一个 Promise。这样五个组件其实是在共享同一个网络结果，真正发出去的请求永远只有一个。

### 1、reactive和ref有什么区别？它们的应用场景是什么，具体是怎么使用的

面试官您好。`ref` 和 `reactive` 都是 Vue 3中用来定义响应式数据的核心 API，但它们在**底层原理、支持的数据类型以及访问方式**上有显著的区别。

**区别**

**数据类型限制**：`ref` 是全能选手，它既可以接收基本数据类型（如 String、Number、Boolean），也可以接收对象和数组。而 `reactive` 它只能接收引用数据类型（Object、Array、Map 等），如果传基本类型给它，会直接报错或失去响应式。

**底层原理**：对于基本类型，`ref` 底层是通过 `Object.defineProperty` 的 `getter` 和 `setter` 来劫持 `.value` 属性实现的；如果传给 `ref` 的是一个对象，它底层其实还是会调用 `reactive`。而 `reactive` 则是纯粹基于 ES6 的 `Proxy` 实现对整个对象的深度代理。

**访问方式**：在 JavaScript 代码中修改或读取 `ref` 的值，必须带上 **`.value`** 后缀（在 template 模板中不需要，Vue 会自动解包）。而 `reactive` 定义的对象，直接像普通 JS 对象一样点（`.`）属性名访问即可，不需要 `.value`。

**各自的应用场景**

- **我会优先使用 `ref` 的场景**：所有独立的、单一的基本数据类型（比如控制弹窗显隐的 `isVisible`，或者一个计数器 `count`）；还有就是当我们需要经常**整体替换**一个数据源时（比如从接口拉取了一个全新的数组重新赋值），用 `ref` 最合适，因为直接修改 `.value` 不会破坏响应式。另外，获取 DOM 元素引用也必须用 `ref`。
- **我会优先使用 `reactive` 的场景**：当有**多个高度相关的属性需要内聚**时，比如一个包含十几个字段的表单对象（账号、密码、手机号），或者复杂的查询参数配置。把它们用 `reactive` 聚合成一个对象，代码语义更清晰，也省去了满屏写 `.value` 的繁琐，非常贴近真实业务的面向对象思维。

总结来说，单一值和需要整体替换的数据用 `ref`，结构化、高内聚的对象用 `reactive`。"

| **对比维度**        | **ref**                                                      | **reactive**                                   |
| ------------------- | ------------------------------------------------------------ | ---------------------------------------------- |
| **支持的数据类型**  | **全部类型**（基本类型 + 引用类型）                          | **仅限引用类型**（Object、Array、Map、Set 等） |
| **JS 中的访问方式** | 必须加 **`.value`**                                          | 直接访问属性（如 `obj.name`）                  |
| **底层核心原理**    | `getter/setter` 拦截 `.value` *(若是对象则内部调用 reactive)* | ES6 **`Proxy`** 深度代理对象本身               |
| **解构赋值**        | 本身就是一个包裹对象，很少解构它                             | **会丢失响应式**（必须套一层 `toRefs` 抢救）   |
| **整体重新赋值**    | **完美支持**（直接覆盖 `.value` 即可）                       | **不支持**（直接重新赋值会切断 Proxy 代理链）  |
| **最爽的应用场景**  | 单一变量、随时需要被整个替换的数据（如接口返回的 List）、DOM 节点引用 | 逻辑高度关联的复杂表单对象、大型配置项对象     |



### 2、Vue3.0的设计目标是什么？相较于Vue2做了哪些优化

**第一，是编程范式与逻辑复用的巨大演进。** Vue 2 采用的是 **Options API（选项式 API）**，数据（data）、方法（methods）、计算属性（computed）是强制分离的。这导致在复杂组件中，同一个业务功能的代码会被拆得七零八落，极难维护。并且 Vue 2 逻辑复用主要靠 Mixin，容易产生命名冲突和数据来源不清晰的问题。 而 Vue 3 全面拥抱了 **Composition API（组合式 API）**。它允许我们将同一个业务逻辑相关的代码高内聚地写在一起，并且可以通过抽离自定义 Hooks 的方式，实现极其优雅、无副作用的逻辑复用。

**第二，是底层响应式系统的重构。** Vue 2 的响应式核心是 `Object.defineProperty`。它的致命缺陷是只能劫持对象已有的属性，无法监听到对象属性的新增和删除，也无法完美拦截数组的索引操作，导致我们经常要用 `$set` 这种补丁 API。 Vue 3 则彻底拥抱了 ES6 的 **`Proxy`**。Proxy 可以直接代理整个对象，拦截几乎所有的对象操作（包含属性的新增/删除、数组索引修改、甚至 Map 和 Set 等数据结构），不仅解决了 Vue 2 的痛点，而且初始化时的性能开销也大幅降低。

**第三，是架构层面的性能优化与工程化支持。** Vue 3 的源码使用 TypeScript 进行了彻底重写，对 TS 的类型推导支持达到了完美的级别。在编译层面，Vue 3 引入了静态提升、事件监听缓存等机制，并且支持了多根节点的 Fragment，减少了无意义的 DOM 层级。在构建工具上，Vue 3 通常与 Vite 结合使用，利用浏览器原生的 ES Module 特性，带来了极速的冷启动和热更新体验，这是以往 Webpack 时代无法比拟的。"

| **对比维度**        | **Vue 2.x**                  | **Vue 3.x**                      | **核心痛点解决 / 优势提升**                                  |
| ------------------- | ---------------------------- | -------------------------------- | ------------------------------------------------------------ |
| **代码组织范式**    | **Options API** (选项式 API) | **Composition API** (组合式 API) | 解决复杂组件代码过于分散（面条代码）的问题，实现高内聚。     |
| **响应式底层**      | `Object.defineProperty`      | **`Proxy`**                      | 解决无法监听属性新增/删除、数组按索引修改的问题，废弃 `$set`。 |
| **逻辑复用方案**    | **Mixins**                   | **自定义 Hooks** (组合函数)      | 解决 Mixin 带来的命名冲突和数据来源不透明问题。              |
| **TypeScript 支持** | 较差 (依赖非官方装饰器)      | **极佳** (源码全量采用 TS 重写)  | 类型推导完美，适合大型企业级项目的持续迭代。                 |
| **多根节点模板**    | 必须包裹唯一的根节点 `<div>` | 支持 **Fragment** (多根节点)     | 减少不必要的 DOM 嵌套层级，提升渲染性能。                    |
| **主流构建生态**    | Vue CLI + Webpack            | **Vite** (或主流 Rollup)         | 基于 ESM，实现极速的毫秒级本地冷启动和 HMR 热更新。          |

**防追问："你刚才提到 Vue 2 的 Mixin 有问题，能具体说说 Mixin 到底坑在哪里吗？为什么 Vue 3 的 Hooks 更好？"**

**回答：**

> "Vue 2 的 Mixin 主要有两大致命痛点：
>
> 1. **命名冲突（隐式覆盖）**：如果引入了多个 Mixin，而它们内部定义了同名的变量或方法，后引入的会悄无声息地覆盖前面的，排查 Bug 极其困难。
> 2. **数据来源不透明（黑盒效应）**：在组件的 template 里看到一个变量，如果它不是当前组件 data 里定义的，你就得去挨个翻那几个 Mixin 文件，才知道它到底是从哪来的。
>
> 而 Vue 3 的 **Hooks（组合函数）** 完美解决了这两点。Hooks 就是普通的纯 JavaScript 函数。我们通过 `const { dataA } = useHookA()` 的显式解构来获取数据，变量来源一目了然；如果发生命名冲突，我们在解构时直接重命名即可（`const { dataA: newDataA } = useHookA()`），彻底消灭了隐式覆盖的风险。"



### 平时项目中用过 Element UI 吧？说说它是怎么和 Vue 进行事件绑定的

"面试官您好。Element UI（或 Element Plus）本质上就是基于 Vue 封装的自定义组件库，因此它与 Vue 的事件绑定完全遵循 Vue 的组件通信机制。

在日常开发中，我主要通过以下三种方式进行事件绑定：

**第一，基础的自定义事件绑定。** 我会使用 `v-on` 指令，或者它的语法糖 `@` 来监听 Element 组件暴露出的事件。比如给 `<el-button>` 绑定 `@click`，或者给 `<el-select>` 绑定 `@change`。在底层，这其实是 Element 组件在其内部的特定时机，通过 `$emit('click')` 或 `$emit('change', value)` 向外派发了事件，而父组件通过 `@` 捕获并执行相应的回调函数。

**第二，表单数据的双向绑定。** 对于 `<el-input>`、`<el-switch>` 这类表单组件，我最常用的是 `v-model` 语法糖。在 Vue 3 中，它本质上是 `:modelValue` 属性绑定和 `@update:modelValue` 事件监听的结合。Element 组件内部在值发生改变时，会 emit 出这个 update 事件，从而实现数据的双向流转。

**第三，获取原生 DOM 事件及 Vue 3 的机制演进。** 如果需要给 Element 组件绑定它并未显式 `emit` 出来的原生事件（比如鼠标原生的 `mouseenter`），在 Vue 2 的时代，我必须使用 `.native` 修饰符（如 `@mouseenter.native`），强制将事件绑定到组件内部的根元素上。 但是在 **Vue 3 中，`.native` 修饰符已经被彻底废弃了**。Vue 3 引入了更加智能的**'透传 Attributes'（Fallthrough）机制**。现在我只需要直接写 `@mouseenter`，只要 Element 组件内部没有通过 `emits` 选项声明这个事件，Vue 就会自动把这个原生事件透传并绑定到组件的根 DOM 节点上。这极大地减轻了开发者的心智负担。"

| --------------------------------------------- | -------------------- | --------------------------------------------- | ------------------------------------------------------------ |
| **监听组件自定义事件**                        | `@change="fn"`       | `@change="fn"`                                | 子组件内部调用 `$emit('change', data)`，父组件捕获。         |
| **表单组件双向绑定**                          | `v-model="val"`      | `v-model="val"`                               | 语法糖。Vue 3 编译为 `:modelValue` 和 `@update:modelValue` 的组合。 |

### 1、说说事件循环

"面试官您好。因为 JavaScript 是一门单线程的语言，为了在执行耗时操作（如网络请求、定时器）时不阻塞主线程，它引入了事件循环机制来实现异步编程。

事件循环的完整运行机制，我认为可以分为以下几个核心步骤：

**第一步：** 执行第一个宏任务（即整个同步代码）。在执行过程中，你可能会注册一些 `setTimeout`（宏任务）和 `Promise.then`（微任务）。

**第二步：** 这个同步代码（第一个宏任务）执行完了，主线程空了。

**第三步：** 此时事件循环会检查**微任务队列**，并一口气把刚才产生的微任务全部跑完。

**第四步：** 微任务跑完了，才去跑下一个宏任务（比如 `setTimeout` 的回调），这样形成一个循环，就是JS的事件循环。

总结来说，核心优先级就是：**执行一个宏任务 -> 清空所有微任务 -> 渲染 -> 再执行下一个宏任务**。"

⚠️注意：整个 `<script>` 标签里的同步代码，其实被浏览器视为"第一个宏任务"



### 2、请说说你对闭包的理解？它在实际开发中有什么缺陷？另外，闭包里的 `this` 指向哪里？

#### 闭包的本质

从底层原理来讲，闭包是**函数和其周围状态的引用捆绑在一起的组合**。在实际项目中，最常见的表达形式就是：**在一个函数内部定义并返回了另一个函数，且内部函数访问了外部函数的局部变量**。闭包的核心作用在于：**保护私有变量、延长变量生命周期**，常用于实现**私有变量封装**和**防抖节流函数**的按需局部状态保存。

#### 闭包中的this指向

在传统的普通函数中，闭包内部函数的 `this` **并不会自动继承外部函数的 `this`**。因为 `this` 的指向是在函数**被调用时**动态决定的，而不是在定义时决定的。当闭包内部函数作为普通函数在全局被独立调用时，它的 `this` 会默认指向全局对象（浏览器中是 `window`，严格模式下是 `undefined`）。 为了解决 `this` 丢失的问题，在 ES5 时代我们通常在外部函数用 `const self = this` 来缓存上下文；但在现在的 ES6 开发中，我会直接使用**箭头函数**来编写内部函数，因为箭头函数没有自己的 `this`，它会完美继承外部词法作用域的 `this`。

#### 闭包的缺陷

闭包最大的副作用是**极易引发内存泄漏**。由于内部函数保留了对外部函数变量的作用域链引用，导致外部函数执行完毕后，其执行上下文中的变量无法被 JavaScript 的垃圾回收机制（GC）销毁。如果闭包被滥用且长期驻留在内存中，会导致页面性能下降甚至卡顿。 我的解决策略是：在不需要使用闭包时，手动将引用该闭包的变量赋值为 `null`，主动断开引用，释放内存。



### 3、小程序上传体积过大如何解决

**第一步，也是见效最快的核心方案：实施分包加载（Subpackages）。** 很多新手会把所有页面都塞在主包里。我会重新规划路由架构，**主包只保留最核心的 TabBar 页面、全局配置以及高频复用的公共组件**。将其他的独立业务线全部拆分到各个'分包'中。按需加载分包，能瞬间把主包体积降下来。同时，为了保证用户体验，我会配合配置**'分包预下载'**，在用户闲时提前静默下载下一个分包。

**第二步，静态资源的全面云端化。** 小程序的本地代码包里，绝对不能存放哪怕一张高清大图或视频文件。我会把项目中所有的背景图、大尺寸 Icon、甚至是自定义字体文件，全部上传到公司专属的 CDN 或 OSS 对象存储上，代码里全部替换为网络 URL。本地只保留几 KB 极其核心的 Base64 占位小图标。

**第三步，依赖库瘦身与代码构建优化。** 很多时候体积膨胀是因为引入了庞大的第三方库。比如处理时间，我会用轻量级的 `Day.js` 替换掉庞大的 `Moment.js`；如图表库，我会使用按需引入的方式打包 ECharts 而不是全量引入。最后，在微信开发者工具中，我一定会勾选**'上传代码时自动压缩'**、**'过滤无用文件'**以及开启依赖分析（Tree Shaking），剔除掉项目里的 Dead Code（死代码）。

总结来说就是：**架构上做分包，资源上上 CDN，代码上精简依赖。**"

### 2、在你的低代码平台项目中，Pinia 是如何驱动数据流转的？

我在 Store 里维护了整个JSON 数组。它不仅仅是一个简单的数组，而是一个包含嵌套关系的 JSON 对象。每一个节点都记录了组件的 ID、类型、属性、样式和事件定义。除此之外，我还维护了一个全局的 `activeId`，用来标记当前用户正在操作哪一个具体的组件。

**其次是典型的'单向数据流'流转过程。** 当用户在属性面板修改一个按钮的背景色时，属性面板**绝不直接去改组件**。它会调用 Store 里的一个 `updateComponent` Action。这个 Action 像是一个严格的安检员，它会根据 `activeId` 找到树里的对应节点，精准修改数据。 重点来了：因为 Vue 3 的响应式机制，中间的画布区域和左侧的大纲树都在'侦听'这棵树。只要 Action 一改完，画布里的组件就会立刻'感知'到属性变了，从而自动触发重绘。



### 3、详细说说 shallowRef 和 shallowReactive 的区别

普通的 `ref` 和 `reactive` 就像是极其负责任的'居委会大妈'。只要你交给它们一个对象，哪怕这个对象里面嵌套了 10 层，它们也会不辞辛劳地拿着 Proxy，一层一层地进去把所有属性都劫持一遍。这就是深度响应式。好处是随便你怎么改里面的深层数据，页面都会刷新；坏处就是，如果这个对象极其庞大（比如我们低代码平台几千行的 JSON 树，或者一个 ECharts 的复杂图表实例），光是初始化这些 Proxy 就能把浏览器卡死。

**为了解决这个性能黑洞，尤雨溪给我们提供了 shallow（浅层）系列。**

**先说 `shallowRef`：** 它是个'只看大门'的保安。它**只关心最外层那个 `.value` 的地址有没有变**。 如果你用它存了一个对象，然后你去悄悄改对象里面的某个深层属性（比如 `state.value.config.color = 'red'`），对不起，Vue 完全装作看不见，页面绝对不更新。 只有当你把整个 `.value` 换成一个全新的对象（比如 `state.value = newObj`），它才会通报说'数据变了，赶紧刷新'。所以它非常适合用来存那些**'要么不改，要改就全量替换'**的大型数据。

**再说 `shallowReactive`：** 它比 `shallowRef` 稍微多管点闲事，它会**劫持对象的第一层属性**。 假设有个对象 `{ a: 1, b: { c: 2 } }`。你改第一层的 `a`，页面会更新；但如果你去改第二层的 `c`，它就不管了。通常在处理那种只有第一层需要动态绑定的扁平化业务数据时，用它能省下不少性能。

**总结一句话就是：** 在复杂的业务里，特别是低代码这种数据爆炸的场景，**绝对不能无脑用普通的 `ref` 和 `reactive`**。遇到第三方大库的实例、庞大的配置大对象，只要确认不需要监听深层变化，一律用 `shallow` 系列，内存占用能瞬间下降一个数量级。"



### 4、原型链机制及 ES5/ES6 继承的底层实现

#### 1. 原型链查找机制（Prototype Chain Resolution）

- **底层数据结构：** 在 JavaScript 中，每个对象内部都维护着一个隐藏的内部插槽 `[[Prototype]]`（可通过 `Object.getPrototypeOf()` 或非标准的 `__proto__` 访问），它指向该对象的构造函数的 `prototype` 属性。
- **查找算法：** 当访问对象实例的某个属性或方法时，JavaScript 引擎首先在实例自身的属性（Instance Properties）中进行查找。若未命中，引擎会沿着 `[[Prototype]]` 指针向上层作用域（即父类原型对象）进行委托查找。此递归查找过程会一直持续，直到命中目标属性，或达到原型链的顶端（`Object.prototype.__proto__` 为 `null`），最终返回 `undefined`。

#### 2. ES5 之前的继承模式（以"寄生组合式继承"为最佳实践）

在 ES6 之前，由于缺乏类的语法糖，开发者需要手动操作原型链。传统的"原型链继承"存在引用类型属性被所有实例共享的内存污染问题，而"借用构造函数"则无法实现原型方法的复用。 业界标准的严谨实现为**寄生组合式继承（Parasitic Combination Inheritance）**，其核心范式分为两步：

- **实例属性继承（构造函数借用）：** 在子类构造函数内部，通过 `Parent.call(this)` 调用父类构造函数。此操作的作用域绑定（Binding）确保了父类中定义的实例属性被独立分配给子类的当前实例，实现了数据的隔离。
- **原型方法继承（原型委托）：** 使用 `Object.create(Parent.prototype)` 创建一个以父类原型为基础的纯净对象，并将其赋值给子类的原型（`Child.prototype`）。此方法避免了直接实例化父类（`new Parent()`）带来的冗余副作用，同时建立了一条干净的 `[[Prototype]]` 委托链。最后需修复子类原型的构造函数指向：`Child.prototype.constructor = Child`。

#### 3. ES6 的 `class` 与 `extends` 机制

- **本质：** ES6 引入的 `class` 关键字在架构层面是寄生组合式继承的**语法糖（Syntactic Sugar）**，并未改变 JavaScript 基于原型委托的底层运行时机制。
- **规范增强：** 虽然是语法糖，但 ES6 规范对其进行了严谨性增强。例如：`class` 声明的所有方法默认是不可枚举的（non-enumerable）；必须使用 `new` 关键字调用；子类构造函数必须通过 `super()` 调用父类，且在调用 `super()` 之前，子类无法访问 `this` 上下文。这确保了父类实例的完整初始化先于子类发生。



### 5、webpack和vite的区别以及各自的优劣势

**Webpack 的工作逻辑是"全量构建"。** 当你运行 `npm run dev` 时，Webpack 必须根据入口文件（Entry），递归地抓取整个项目的所有模块（包括你的业务代码和上千个 npm 第三方包），构建出一张完整的**依赖关系图（Dependency Graph）**。在内存中完成所有的编译、转译和打包（Bundle）操作后，才启动开发服务器。

- **劣势**：随着项目规模（Module Count）增加，构建时间呈线性甚至指数级增长。
- **结果**：大型项目冷启动可能需要数十秒甚至分钟级。

**Vite 的工作逻辑是"双轨制"处理。**

- **第三方依赖（Dependencies）**：对于那些几乎不会变动的 npm 包，Vite 在启动前会进行一次性**预构建（Pre-bundling）**。
- **业务代码（Source Code）**：Vite 利用浏览器原生的 **ES Modules (ESM)** 规范。它不进行全量打包，而是直接启动开发服务器。浏览器通过 HTTP 请求按需获取模块，Vite 仅对当前请求的单个模块进行即时转换（On-demand transformation）。
- **结果**：冷启动时间与项目规模脱钩，始终保持在毫秒级。

#### 2. 热更新（HMR）的效率瓶颈

**Webpack 的 HMR 复杂度随项目增大而降低。** 在 Webpack 模式下，当你修改一个文件，即使有热更新机制，它依然需要重新构建受影响的整个依赖路径（Chunk）。虽然有缓存机制，但在数千个模块的应用中，重新生成 Bundle 段并推送到浏览器的过程依然会有明显的延迟。

**Vite 的 HMR 复杂度始终为 O(1)。** 由于 Vite 是基于浏览器原生的 ESM，当某个文件被修改时，Vite 只需要让该模块在浏览器中的缓存失效。浏览器会重新发起该特定模块的 HTTP 请求，Vite 只需重新编译这一个文件。这种"点对点"的精确更新，使得热更新速度不受项目体量影响，永远保持瞬时反馈。

#### 3. 底层工具链的语言性能压制

**Webpack 依赖 JavaScript 实现。** Webpack 的大部分处理逻辑（包括各种 Loaders 和 Plugins）都是运行在 Node.js 环境下的 JavaScript 代码。受限于 JS 的单线程特性和解释型语言的性能上限，其处理大规模 AST（抽象语法树）转换的速度存在物理瓶颈。

**Vite 依赖 Go 语言实现（esbuild）。** Vite 在处理第三方依赖的"预构建"阶段，使用的是用 **Go 语言** 编写的 **esbuild**。

- **性能差距**：Go 是编译型语言，支持原生的多线程并发。esbuild 的解析、编译速度比基于 JS 的打包工具快 **10 到 100 倍**。这使得 Vite 能够极速处理那些庞大的第三方依赖库。

| **维度**       | **Webpack (Bundle-based)**   | **Vite (Native ESM-based)**    |
| -------------- | ---------------------------- | ------------------------------ |
| **冷启动**     | 极慢（需要全量打包后再启动） | 极快（直接启动，按需编译）     |
| **热更新**     | 随规模变大而变慢             | 瞬时完成（不受规模影响）       |
| **底层工具**   | Node.js (JavaScript)         | esbuild (Go) + Rollup (JS)     |
| **配置复杂度** | 高（Loader/Plugin 繁琐）     | 低（开销即用，配置极简）       |
| **浏览器支持** | 兼容性极强（支持旧版）       | 本地开发要求支持原生 ESM       |
| **成熟度**     | 极其成熟，复杂插件生态完善   | 较新，部分极其冷门的插件需适配 |

# Vue 面试题集锦

## 1、什么是虚拟DOM，如何实现一个虚拟DOM

"面试官您好。关于虚拟 DOM，它本质上是一个用来描述真实 DOM 结构的轻量级 JavaScript 对象。在早期没有虚拟 DOM 的时代，前端开发往往需要手动用原生 API 去频繁操作真实 DOM。由于真实 DOM 节点上挂载了极其庞杂的属性，且每次修改都极易引发浏览器昂贵的重排与重绘，这在处理复杂交互时不仅代码难以维护，性能也很容易失控。虚拟 DOM 的出现就是为了在真实 DOM 和 JavaScript 逻辑之间搭建一个桥梁。当我们页面中的数据状态发生变化时，Vue 等框架并不会立刻去触碰真实的页面，而是先在内存中快速生成一棵全新的虚拟 DOM 树。接着，框架会利用核心的 Diff 算法，将这棵新树与上一次渲染的旧树进行同层级的精细比对，找出两者之间真正发生变化的最小差异点。最后，框架只会把这些经过精准计算的差异，一次性地更新到真实的 DOM 上，避免了频繁操作真实 DOM 所引发的性能问题。在像表单设计器这种充斥着大量组件拖拽、属性联动和高频状态更新的复杂系统中，虚拟 DOM 的机制能够自动将多次数据修改合并为一次视图更新，它不仅极大地解放了开发者的心智，让我们能纯粹聚焦于数据状态的管理，同时也在绝大多数场景下，为页面的渲染性能提供了一个非常可靠的下限保障。"

### 虚拟DOM长什么样？

真实的 DOM 节点（哪怕是一个空的 `<div>`）打印出来都有几百个原生属性。而虚拟 DOM（通常被称为 VNode）只是一个我们自己定义的、非常纯粹的 JS 对象。比如你在模板里写了 `<div class="container">Hello</div>`，在 Vue 底层它会被转换成类似这样的普通对象：

```js
// 操作这种纯 JS 对象的开销是极小的，无论你怎么在内存里修改它，都不会引发浏览器的渲染引擎工作。
const vnode = {
  tag: 'div',              // 标签名
  props: {                 // 属性节点
    className: 'container'
  },
  children: ['Hello']      // 子节点（纯文本或其他 VNode）
}
```

### 如何实现虚拟DOM

分为三步，第一步是要构造一个对象模型，它接收标签名、属性字典以及子节点序列作为参数，最终返回一个纯净的 JavaScript 树形结构对象，也就是常说的 VNode。第二步是实现"首次渲染引擎"，也就是编写一个 render 函数，它负责递归地遍历这个 VNode 对象，调用底层的 `document.createElement` 和 `setAttribute` 等原生 API，将虚拟节点转化为真实的 DOM 节点，并挂载到页面的物理容器中。第三步，也是最核心的一步，即实现"Diff 与补丁更新机制"，我会编写一个 `patch` 函数，当页面的数据状态发生改变、生成了全新的 VNode 树时，这个函数会同步接收旧的 VNode 和新的 VNode。为了保证性能，它会采取同层比较的启发式策略，严格对比节点的标签类型、文本内容以及子节点数组。如果发现标签类型改变，直接暴力卸载重建；如果仅仅是属性变化，则精准调用原生 API 更新对应的属性；如果是复杂的列表子节点变化，则会结合 `key` 属性通过双端指针比对等算法，计算出开销最小的真实 DOM 移动、插入或删除操作。

## 2、说说对双向绑定的理解

"面试官您好。我对双向数据绑定的理解，本质上是框架为了极致简化表单交互和状态同步而提供的一颗'语法糖'。所谓'双向'，包含了两条数据流向：第一条是'数据到视图'（Model to View），当内存中的数据状态发生改变时，框架的响应式系统会自动劫持这些变化，并驱动虚拟 DOM 重新渲染，最终更新页面；第二条是'视图到数据'（View to Model），当用户在页面上进行输入或交互时，框架底层通过自动绑定的 DOM 事件监听器（比如 `input` 或 `change` 事件），捕获用户的输入值，并自动将其反写回内存中的数据模型。以 Vue 中的 `v-model` 为例，它在底层其实就是 `v-bind:value`（绑定数据）和 `v-on:input`（监听输入事件）的结合体。通过这种机制，我们彻底告别了频繁的 `document.getElementById` 和 `addEventListener`，让开发者能够纯粹地聚焦于底层业务数据的流转。当然，在底层实现上，这套机制强依赖于框架的响应式引擎，比如 Vue 2 使用的 `Object.defineProperty` 以及 Vue 3 升级后的 `Proxy` 数据劫持方案。"

### 底层原理

#### **第一向：视图驱动数据 (View → Model)**

这一向比较简单，属于浏览器的原生能力。

- **底层机制**：框架在编译模板（Template）时，一旦解析到 `v-model` 指令，就会自动给当前的 DOM 元素生成一段事件监听代码。
- **物理动作**：对于原生的 `<input type="text">`，框架会自动挂载 `addEventListener('input', (e) => { data.value = e.target.value })`。用户一敲键盘，DOM 事件触发，立刻把新值塞回 JavaScript 变量中。

#### **第二向：数据驱动视图 (Model → View) —— 绝对的核心考点**

这一向极其复杂，是所有现代响应式框架的心脏。它需要解决一个问题：**JS 变量变了，怎么通知页面更新？**

- **数据劫持（观察者机制）**：
  - **Vue 2 的痛点方案**：利用 `Object.defineProperty`，递归遍历 data 对象的所有属性，为它们重写 `getter` 和 `setter`。你读取数据触发 getter（收集依赖），你修改数据触发 setter（通知更新）。
  - **Vue 3 的降维打击**：利用 ES6 的 `Proxy`。它不再是劫持单个属性，而是直接给整个对象套上一层"代理层"。不管你是新增属性、删除属性，还是通过索引修改数组，代理层都能完美拦截，性能和覆盖面实现了质的飞跃。
- **派发更新**：一旦 setter 或 Proxy 的 set 夹子被触发，它会立刻通知对应的 Watcher（观察者），Watcher 接着触发我们在上一题聊过的虚拟 DOM 的 `render` 和 `patch` 流程，最终把新数据刷到屏幕上。

关于原生浏览器的渲染机制，我们必须明确一个核心边界：'数据驱动视图'绝对不是浏览器的原生能力，它是现代前端框架（如 Vue/React）用极其复杂的底层代码人工制造出来的一层'幻觉'。在最纯粹的原生 JavaScript 中，如果你定义了一个变量 `let name = "张三"`，然后在代码里把它改成 `"李四"`，浏览器屏幕上的任何 DOM 元素都绝对不会有任何反应。原生浏览器只认识 DOM API，你要改变视图，必须手动去执行 `document.getElementById('name').innerText = "李四"`，这就是传统的'DOM 驱动'或'过程式编程'。现代框架之所以能做到'数据驱动视图'，是因为它们在底层劫持了数据（通过 Proxy 或 defineProperty），当察觉到数据变化时，是框架自己偷偷帮我们执行了那些繁琐的 DOM 操作。

而反过来，为什么我说'视图驱动数据'属于原生能力？因为当你对着一个原生的 `<input>` 输入框敲击键盘时，浏览器原生的渲染引擎会立刻把输入的字符显示在屏幕上（视图已经改变），同时原生触发一个 `input` 事件。我们只需要用最基础的原生 API `addEventListener` 监听这个事件，就能轻松拿到用户输入的值并赋给 JS 变量。所以，'视图层的交互反馈并将数据抛给 JS'是浏览器天生自带的机制，而'JS 数据变了自动去改视图'则是框架革命性的伟大发明。

## 3、Vue中的v-show和v-if怎么理解？

"面试官您好。在 Vue 中，`v-if` 和 `v-show` 虽然都能控制元素的显隐，但它们的底层原理和性能消耗侧重点完全不同。`v-if` 是'真正'的条件渲染，它是控制 DOM 节点存在与否的开关。当条件为假时，Vue 会直接在真实 DOM 树中销毁该节点及其内部的所有子组件和事件监听器；当条件为真时，再重新创建并挂载真实 DOM。这种机制导致它的初始渲染开销较小（如果初始为假，根本就不渲染），但在频繁切换时，会引发极大的 DOM 重排和重绘性能损耗。

相对而言，`v-show` 则是简单的 CSS 切换。无论初始条件真假，元素都会被立刻渲染到真实的 DOM 树中。当条件为假时，Vue 仅仅是通过给元素添加内联样式 `display: none` 来达到视觉上的隐藏。因此，`v-show` 的初始渲染开销较大，但后续状态切换时的性能极高，因为它仅仅触发了浏览器的重绘，完全避开了昂贵的 DOM 节点创建与销毁过程。在工程实践中，我们会根据组件切换的频率来做技术选型：频繁切换的场景优先使用 `v-show`，而条件很少改变或涉及大量子组件的场景则优先使用 `v-if`。"

| **特性维度**     | **v-if (条件渲染)**                                          | **v-show (CSS 切换)**                                        |
| ---------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **底层原理**     | 动态地向 DOM 树中**添加或移除**真实的 DOM 节点。             | 无论真假都保留在 DOM 树中，仅仅切换内联样式 **`display: none`**。 |
| **生命周期触发** | 切换时会触发子组件的完整生命周期（`created`, `mounted`, `destroyed` 等）。 | 切换时**完全不触发**组件的创建和销毁生命周期。               |
| **初始渲染开销** | **低**。如果初始条件为 `false`，它什么都不做，连内部的图片都不会去加载。 | **高**。即便初始条件为 `false`，也会在底层全部渲染完毕，只是视觉上不可见。 |
| **频繁切换开销** | **极高**。每次切换都要执行昂贵的 DOM 增删操作，引发全页面的重排。 | **极低**。仅仅修改 CSS 样式，属于轻量级的操作。              |
| **搭配特性**     | 可以配合 `v-else-if` 和 `v-else` 使用，也可以用在 `<template>` 包装标签上。 | 只能独立使用，且**不支持** `<template>` 语法。               |

## 4、说说对SPA的理解

SPA（Single Page Application，单页应用）是一种网络应用程序的架构模式。从浏览器解析的物理层面来看，**SPA 在其完整的生命周期中，只会向服务器请求唯一的一个完整 HTML 文档（通常是一个只包含 `<div id="app"></div>` 的空壳）。** 在后续的所有用户交互和页面跳转中，浏览器**绝对不会**发生原生的全局刷新（Full Page Reload）。SPA 的运行完全依赖于客户端 JavaScript 引擎接管浏览器的路由行为。当用户点击链接切换页面时，前端路由机制会拦截这一默认的页面跳转请求，利用 AJAX 或 Fetch API 仅向服务器请求纯粹的业务数据（如 JSON），随后通过 JavaScript 动态地销毁和创建真实的 DOM 节点，从而在当前唯一的页面壳子内完成局部视图的替换。这种机制本质上是将传统由服务器完成的"HTML 拼接与渲染"工作，全部转移到了客户端浏览器的内存中执行，也就是 CSR（客户端渲染）。

| **维度**           | **SPA 的绝对优势**                                           | **SPA 的致命劣势 (代价)**                                    |
| ------------------ | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **渲染性能与 UX**  | **无缝的沉浸式体验**。页面切换无需等待服务器返回整页 HTML，没有白屏闪烁，完全媲美原生 App（Native App）的流畅度。 | **首屏加载时间（FCP）极度缓慢**。初始访问时，浏览器必须下载完整庞大的框架源码（如 Vue/React 运行时）和业务打包后的 JS bundle，解析完毕后才能渲染出首屏内容。 |
| **架构与开发协同** | **彻底的前后端分离**。后端纯粹化，退化为只提供 RESTful 或 GraphQL API 的数据提供者；前端完全掌控视图层和路由架构。 | **内存管理要求极高**。由于页面从未真正刷新，如果 JS 代码存在内存泄漏（如未解绑的全局事件监听器），随着用户操作时间的延长，浏览器内存会被逐渐吃满，最终导致页面卡死。 |
| **网络生态环境**   | 组件状态可以在页面切换间完美保留（如音乐播放器在后台播放时切换歌单页面，音乐不会中断）。 | **SEO（搜索引擎优化）极其困难**。传统的搜索引擎爬虫抓取到的只是一个没有任何内容的 HTML 空壳，导致页面极难被收录（除非使用专门的预渲染或 SSR 技术进行弥补）。 |

| **架构维度**           | **单页应用 (SPA - Single Page Application)**                 | **多页应用 (MPA - Multi Page Application)**                  |
| ---------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **核心组成**           | 唯一的 `index.html` + 庞大的 JS Bundle + 业务组件。          | 多个物理存在的完整 `.html` 文件。                            |
| **路由控制权**         | **前端主导**（基于 `hashchange` 或 HTML5 `History API` 劫持跳转）。 | **后端主导**（传统的超链接 `href`，直接向服务器请求新路径）。 |
| **页面刷新机制**       | **局部动态重绘**。页面不刷新，浏览器原生的 `window` 和 `document` 对象始终存活。 | **全局强刷**。每次跳转都彻底销毁旧文档，重新下载并解析新文档。 |
| **渲染计算位置**       | **客户端 (浏览器)**。JS 在用户设备上执行计算，动态生成 DOM 节点。 | **服务端 (服务器)**。服务器拼接好完整的 HTML 字符串后直接下发给浏览器。 |
| **数据传输载荷**       | 只有首次请求下发 HTML 壳，后续路由切换**仅传输纯净的业务数据 (JSON)**。 | 每次路由切换都必须**重新传输包含布局结构的完整 HTML 文档**。 |
| **首屏加载 (FCP)**     | **极慢**。必须等待庞大的框架运行时 JS 下载并解析完毕，才能开始渲染首屏 DOM。 | **极快**。浏览器收到包含 DOM 结构的 HTML 字符串后立刻触发渲染管线。 |
| **后续交互性能**       | **极速流畅 (沉浸感)**。由于资源已缓存且仅更新局部 DOM，体验媲美原生 App。 | **割裂感强**。每次点击跳转必定伴随短暂白屏和网络加载。       |
| **内存与状态管理**     | **极其脆弱**。状态可以在页面间轻松共享（如 Vuex/Redux），但极易发生内存泄漏。 | **天生安全**。每次跳转内存全部清空重建，绝不存在跨页面的 JS 内存泄漏。 |
| **SEO (搜索引擎优化)** | **极差**。爬虫抓取到的通常是一个只包含 `<div id="app"></div>` 的空壳。 | **极佳**。每个页面都有完整且语义化的 HTML 标签，爬虫可直接解析文本。 |

## 5、SPA应用首屏加载速度慢怎么解决？

"面试官您好，首先我们要明确'首屏加载速度'是指**浏览器从响应用户输入网址地址，到首页内容渲染完成的时间**。而导致它加载速度慢的原因有以下几点：1、网络延迟问题 2、资源文件体积是否过大 3、资源是否重复发送和加载 4、加载脚本的时候，渲染内容阻塞了。所以解决加载速度慢的问题本质上就是解决网络延迟、压缩资源以及对抗浏览器 JS 解析主线程阻塞。解决的方法也就围绕着这些方面展开，首先是网络延迟层面，通过强制开启服务器的 Brotli 或 Gzip 压缩算法，配合静态资源的 CDN 边缘节点分发，以及 HTTP/2 的多路复用机制，从物理链路上极速压榨文件的下载时间，尽量降低网络延迟对加载速度的影响。其次是我会利用 Webpack 或 Vite 的底层机制执行极致的代码分割。将庞大的第三方依赖库（Vendor）单独抽离缓存，并严格实施路由级别的懒加载，确保首屏绝对不加载非首屏的 JS 代码。最后是针对极其复杂的业务组件实施异步按需加载，或者在极端的首屏指标要求下，引入骨架屏（Skeleton Screen）甚至预渲染（Prerendering）技术，在 JS 引擎彻底接管页面前，预先向浏览器输出静态的 HTML 结构，从而实现视觉上的极速响应。"

| 如何排查                                                     | 解决方案                          | **靶向治疗方案 (物理层级的降维打击)**                        |
| ------------------------------------------------------------ | --------------------------------- | ------------------------------------------------------------ |
| 过滤 JS 和 CSS 请求，查看耗时最长的文件。如果蓝色长条（Content Download）极长，说明文件太大或未开启压缩；如果绿色长条（Waiting for TTFB）极长，说明服务器响应慢或没有命中 CDN。 | **病因 A：单个 JS 文件体积超大**  | 1. **路由懒加载**：使用 `const Home = () => import('./Home.vue')`，将代码按页面物理切分。 2. **第三方库外包**：配置 `splitChunks`，将 Vue、Vue-Router 剥离为独立的 vendor 包，利用浏览器强缓存。 3. **按需引入**：坚决干掉全量引入，UI 组件库和图表库严格使用按需打包插件。 |
| 在本地运行分析插件，它会生成一张可视化的矩形树图。你可以一眼看出哪个第三方库（比如 Moment.js 或 ECharts）占用了不合理的体积，或者哪些本该拆分的路由被错误地打包在了一起。 | **病因 B：文件下载极其缓慢**      | 1. **开启传输压缩**：服务端强制开启 Gzip 甚至更高压缩比的 Brotli，体积瞬间缩小 70% 以上。 2. **CDN 边缘分发**：将静态资源部署到 CDN，让用户从地理位置最近的节点拉取代码。 3. **HTTP/2 多路复用**：解决 HTTP/1.1 的队头阻塞问题，允许并发下载大量碎小文件。 |
| 录制首屏加载过程，查看主线程（Main 轨道）中是否存在右上角标红的"Long Task"（执行时间超过 50ms 的长任务）。这能精准定位到是哪一段 JS 逻辑把浏览器卡死了。 | **病因 C：JS 执行导致长时间白屏** | 1. **骨架屏 (Skeleton)**：在 HTML 壳子里写死极简的 CSS 占位图，安抚用户心理预期。 2. **服务端渲染 (SSR) / 预渲染**：如果项目对首屏要求极高，直接改用 Nuxt 架构，让服务器吐出带有完整 DOM 的 HTML。 3. **时间切片**：如果首屏必须计算海量数据，使用 `requestAnimationFrame` 将长任务拆解到多个渲染帧中执行。 |

## 6、说说你对Vue生命周期的理解

"面试官您好。从底层架构的视角来看，生命周期本质上是 Vue 组件实例从'内存空间开辟'到'物理 DOM 挂载'，再到'数据驱动视图更新'，最终走向'内存释放与节点物理抹除'的完整流水线。在这条流水线上，Vue 引擎会在特定的物理节点暴露出'钩子函数'（Hooks），赋予开发者在框架执行流中注入自定义逻辑的最高权限。

整个生命周期严格划分为四大物理阶段：**初始化阶段**（确立响应式引擎）、**挂载阶段**（将虚拟 VNode 编译并转译为真实 DOM）、**更新阶段**（拦截数据变动并执行 Diff 算法打补丁）、以及**销毁阶段**（切断所有响应式追踪并剥离物理节点）。在实际的大型工程中，比如开发高度动态化的表单设计器时，我们必须极其严谨地依据这些底层时机来做技术决策。例如：绝对不能在挂载前去操纵 DOM，必须在销毁阶段手动清理自定义的全局事件与定时器以防内存泄漏。掌握生命周期，实质上就是掌握了 Vue 框架的运行心跳。"

| **生命周期钩子**                  | **描述**                                                     | **数据状态 (响应式)**                                      | **DOM 状态 (物理节点)**                                      | **绝对正确的实战场景**                                       |

## 8、为什么data属性是一个函数而不是一个对象？

"面试官您好。在 Vue 的组件化架构中，data 是**用于声明组件初始化响应式状态的函数**。组件的 `data` 属性之所以强制要求是一个函数，而绝对不能是一个纯粹的对象，其根本原因在于 JavaScript 引擎对引用数据类型（Object）的内存分配机制。当我们定义一个 Vue 组件时，本质上是在定义一个可被无限次复用的'蓝图'。如果我们将 `data` 直接写成一个普通对象，由于对象在 JS 中是引用类型，所有基于这个蓝图创建出来的组件实例，在底层都会拿到指向同一块物理内存地址的指针。这会导致极其致命的'数据污染'灾难：只要你在页面上修改了其中一个组件实例的数据，其他所有同类组件的数据都会瞬间被同步篡改。

为了彻底斩断这种内存共享关联，Vue 在底层设计上要求 `data` 必须是一个工厂函数。每当框架需要创建一个新的组件实例时，它就会调用一次这个 `data` 函数。通过函数的执行，在内存中动态生成并返回一个全新的、拥有独立内存地址的对象。这样就从物理层面上保证了无论这个组件被复用多少次，每一个实例都拥有属于自己绝对私有、互不干扰的数据状态。"

| **核心概念**                              | **底层内存形态 (JS 引擎如何处理)**                           | **物理世界比喻 (极简白话)**                                  | **致命实战陷阱 / 框架应用**                                  |
| ----------------------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **基本数据类型** (Number, String等)       | 值直接存储在**栈内存 (Stack)** 中。彼此绝对物理隔离。        | **复印件**。你拿走一份复印件随便改，对我的原件毫无影响。     | 毫无陷阱。随意赋值，天生安全。                               |
| **引用数据类型** (Object, Array等)        | 真实数据存在庞大的**堆内存 (Heap)** 中，变量（栈内存）里只存了一个指向该数据的**内存地址指针**。 | **房间与钥匙**。数据是房间里的家具，变量只是开门的钥匙。     | 极易发生"一处修改，处处灵异同步"的数据污染灾难。             |
| **直接赋值** (`a = b`)                    | 仅仅复制了栈内存中的**地址指针**，没有任何堆内存空间的开辟。 | **配备用钥匙**。多了一把钥匙，但大家开的依然是同一扇门。     | 绝对禁止用这种方式传递需要独立编辑的复杂表单对象。           |

| **深拷贝** (`structuredClone`)            | 递归遍历，在堆内存中**开辟绝对独立的全新空间**，物理结构 1:1 完美克隆。 | **1:1 完美建造的新房与新保险箱**。与原来的宇宙彻底失去联系。 | **终极解法**。在编辑复杂状态前（如打开数据配置弹窗），必须深拷贝原数据作为副本操作。 |
| **Vue 的 `data` 是对象** *(灾难反模式)*   | 所有的组件实例在底层共享了同一个引用对象的内存地址。         | **发备用钥匙**。所有复用该组件的地方，都在操作同一个数据源。 | 面试必考绝对禁区。会导致页面上所有的同类组件数据发生灵异同步。 |
| **Vue 的 `data` 是函数** *(官方强制标准)* | 强制作为"对象工厂"执行。每次实例化组件，函数都会执行并 `return` 一个开辟了全新内存空间的对象。 | **新房建造工厂**。每次调用，都给你盖一栋全新的、带独立钥匙的别墅。 | 完美实现了组件化开发中最核心的"状态私有、互不干扰"原则。     |

## 9、Vue组件间通信方式都有哪些?

"面试官您好。在 Vue 中，组件间通信的本质是解决'**不同组件之间的数据流转与状态同步**'问题。为了保证数据流向的绝对清晰和可追溯，Vue 遵循'单向数据流'的设计哲学。对于最基础的父子组件通信，我们严格遵守 `props` 向下传递数据，`$emit` 向上派发事件的规则。如果遇到嵌套极深的祖孙组件通信，为了避免一层层繁琐地透传 `props`，我会采用 `provide` 与 `inject` 进行依赖注入，实现跨层级的'降维通信'。而对于兄弟组件（比如点击左侧面板，中间画布立刻响应），在轻量级场景下，早期 Vue 2 会利用 Event Bus（事件总线）通过全局发布订阅模式来解决；但在现代复杂工程中，为了保证状态的可预测性和时间旅行调试，我会毫不犹豫地引入 Vuex 或 Pinia 这样的全局集中式状态管理方案。此外，还有像 `$refs` 这种直接获取底层物理实例的越权手段，通常只作为操作 DOM 或调用子组件内部原生方法的最后底牌来使用。"

| **通信通道与 API**                          | **适用的物理层级**        | **底层数据流转机制与绝对红线**                               | **表单设计器实战场景 (范例)**                                |
| ------------------------------------------- | ------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **1. `props` / `$emit`** *(绝对主力)*       | **父子组件** (直接相邻)   | **单向数据流**。父级通过 `props` 下发只读数据，子级绝对禁止直接修改。子级只能通过 `$emit` 抛出自定义事件，通知父级自己去改。 | 父组件（画布）向子组件（输入框控件）传入基础配置数据。       |
| **2. `v-model`** *(语法糖)*                 | **父子组件**              | 本质是 `props: value` 和 `@input` 事件的底层结合体。实现了父子间看似双向绑定的魔法。 | 右侧属性面板组件修改某个字段时，直接双向同步到画布选中的控件上。 |

### provide 与 inject

"当一个祖先组件（比如表单设计器的最外层根容器）使用了 `provide` 选项时，它实际上是在自己的物理内存空间中划出了一块'共享数据区'，并像广播塔一样将这些数据向所有处于其所在组件树分支下的后代组件进行广播。而任何一个后代组件（无论是儿子、孙子还是重孙子），只要使用了 `inject` 选项并声明了对应的接收键名，就相当于打开了特定频道的收音机，能够瞬间无视中间跨越了多少个层级，直接且精准地读取到祖先组件提供的共享数据。这种'依赖注入'的模式，彻底斩断了传统 `props` 必须逐层透传的物理链条，极大地降低了深层嵌套组件的耦合度。"

## 10、说说你对slot的理解？slot的使用场景有哪些？

"面试官您好。我对插槽的理解，本质上它是 Vue 官方提供的一套'内容分发 API'。它的底层物理意义在于：**子组件在其自身的模板中，提前挖好一个或多个物理占位符，而这些占位符里具体要填充什么 DOM 结构，绝对控制权完全交还给父组件。** 这种机制极大地提升了组件的复用率和灵活性。

在实际工程中，插槽严格划分为三个物理层级。第一层是'默认插槽'，用于最基础的单一内容替换；第二层是'具名插槽'，当子组件具备复杂的布局（如弹窗的页头、主体、页脚）时，父组件可以通过指定名称，实施精准的 DOM 定向投放；第三层，也是最核心的'作用域插槽'，它打破了传统的作用域物理隔离，允许子组件将自己内部的私有数据反向暴露给父组件，父组件拿到这些数据后，再结合自身的业务逻辑，动态生成 DOM 结构并塞回子组件的占位符中。在实现诸如动态表格、表单画布控件包装器等底层业务时，作用域插槽是无可替代的绝对核心。"

| **插槽形态**   | **底层物理逻辑**                                             | **编译与作用域归属**                                         | **实战适用场景**                                             |
| -------------- | ------------------------------------------------------------ | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **默认插槽**   | 子组件内写一个 `<slot>` 标签作为唯一坑位。父组件标签内部包含的所有未指定名称的 DOM，都会被一股脑打包塞进这个坑里。 | 父组件模板里的所有内容，都是在**父组件的作用域**内编译完毕后，才作为成品传递给子组件的。 | 极简的包装容器。例如一个带有特定阴影和边框的 `Card` 组件，里面具体放文字还是图片由外部决定。 |
| **具名插槽**   | 子组件内写多个 `<slot name="xxx">`。父组件通过 `<template v-slot:xxx>` 将 DOM 片段精准投递到对应名称的坑位中。 | 同上，依然在父级作用域编译。                                 | 复杂的结构化组件。例如 `Dialog` 对话框组件，天然需要区分标题区、内容区和底部按钮操作区。 |

## 11、v-if和v-for如何使用？为什么不建议同时使用它们两个

`v-if` 指令用于条件性地渲染一块内容。这块内容只会在指令的表达式返回 `true` 值的时候被渲染。

`v-for` 指令基于一个数组来渲染一个列表。`v-for` 指令需要使用 `item in items` 形式的特殊语法，其中 `items` 是源数据数组或者对象，而 `item` 则是被迭代的数组元素的别名。

在 `v-for` 的时候，建议设置 `key` 值，并且保证每个 `key` 值是独一无二的，这便于 Diff 算法进行优化。

```vue
<Modal v-if="isShow" />

<li v-for="item in items" :key="item.id">
    {{ item.label }}
</li>
```

之所以不建议同时使用，核心原因是"**执行的优先级冲突**"，但是在 Vue 2 和 Vue 3 中二者优先级不同，所导致的问题也不同。

**在 Vue 2 中，v-for 的优先级是要高于 v-if 的。**

假设你写了这样一段代码，本意是想渲染列表中状态为 `active` 的用户：

```vue
<li v-for="user in userList" v-if="user.isActive" :key="user.id">
  {{ user.name }}
</li>
```

所产生的问题：

哪怕你的 `userList` 里有 10000 个人，而只有 1 个人是 `isActive` 的，Vue 2 也会**极其死板地先完整循环 10000 次**，生成 10000 个虚拟 DOM 节点，然后在挂载前，再逐一用 `v-if` 去判断。发现前 9999 个都不符合条件，于是把它们全扔掉，只渲染最后那 1 个。

**在 Vue 3 中，"直接报错"（v-if 优先级更高）**

同样的代码，在 Vue 3 中所产生的问题：

因为 `v-if` 拥有了最高优先级，它会**先于** `v-for` 执行。当代码解析到 `v-if="user.isActive"` 时，`v-for` 还没开始跑呢！此时上下文中根本不存在 `user` 这个变量！**结果：** 你的代码不仅不能运行，而且会在控制台直接爆红报错。
